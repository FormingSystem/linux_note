# 第10章_SMP_与中断亲和性_IPI_机制

## 10.1_章节内容说明

本章在第9章“层级中断控制器”之后，转向**多核（SMP）场景下的中断问题**：当同一台 Linux 机器上有多个 CPU 时，一个外设中断究竟应该送到哪颗 CPU？能不能在运行期迁移？为什么 /proc/interrupts 里有时一列数字特别高、别的 CPU 几乎不动？为什么网络驱动、存储驱动都在讲“中断绑核/中断分散”？这些问题都不属于“层级 irq_domain”范畴，而是属于**SMP 环境下的中断亲和性（affinity）与处理器间中断（IPI）**这一类机制。

本章目标：

1. 说明在 Linux 中断子系统里，SMP 会带来哪些新的约束与选择；
2. 说明中断亲和性（irq affinity / smp_affinity）的语义、作用域与配置方式；
3. 说明 IPI 是什么、为什么必须存在、和普通外设中断的本质区别；
4. 给出驱动开发者在“多核 + 高中断量”场景下应当写/不应当写的代码；
5. 给出一套排查“单核中断打满/中断不均衡”问题的步骤。

本章结构：

- 10.1 SMP 场景下中断要解决的核心问题
- 10.2 中断亲和性与 smp_affinity 的内核语义
- 10.3 IPI（处理器间中断）的类型与内核用途
- 10.4 中断分散策略与 irqbalance 的协同
- 10.5 驱动开发者的接口选择与注意事项

## 10.2_SMP_场景下中断要解决的核心问题

### 10.2.1_什么是多核场景下的_中断路由

在单核系统里，中断只可能送到那一颗 CPU；
 在多核系统里，同一个外设的中断可以送到**某一颗** CPU，也可以送到**一组** CPU，还可以在运行期改变要送往的 CPU，这个“送到哪儿”的选择过程就叫**中断路由**。

### 10.2.2_为什么要做中断路由

主要有三类原因：

1. **负载均衡**：如果所有外设中断都送到 CPU0，那么 CPU0 会长期处于中断上下文，用户态和其他内核线程得不到调度，表现就是“系统整体看起来很慢，但其他 CPU 很空”。
2. **缓存/本地性优化**：某些中断（尤其是网络、存储）是在特定 CPU 上有数据路径的，把中断送到那个 CPU，可以减少跨核缓存同步的开销。
3. **隔离/实时性**：有时需要把一部分中断固定在非实时核，留出实时核给特定任务，这时必须能指定“这条中断只能送到哪几个 CPU”。

### 10.2.3_为什么不能_全部广播给所有_CPU

这是很多初学者的误解。把一个外设中断送给所有 CPU，看上去“大家都能处理”，实际上会造成：

- 多个 CPU 同时抢同一把锁 → 反而变慢；
- 同一事件被多次处理 → 产生重复工作；
- 必须在中断入口做额外过滤 → 增加开销。

因此正确的做法是：**为一个中断选择一个（或一组很小的）目标 CPU 集合**，而不是广播式分发。

### 10.2.4_层级中断与_SMP_中断的关系

第9章讲的是“中断从外设一路走到 GIC”的问题；
 第10章讲的是“GIC 收到中断后，要把它送到哪颗 CPU”的问题。
 也就是说：**第9章解决“能不能到 GIC”，第10章解决“到 GIC 以后交给谁”**。两者是连续的，但关注点不同。

## 10.3_中断亲和性与_smp_affinity_的语义

### 10.3.1_中断亲和性是什么

**中断亲和性（IRQ affinity）**就是给某个 IRQ 绑定一组 CPU 掩码，告诉内核：当这个 IRQ 发生时，应当优先（或仅）在这一组 CPU 上处理它。

这组掩码通常可以在以下位置看到/修改：

- `/proc/irq/<irq>/smp_affinity`
- `/proc/irq/<irq>/smp_affinity_list`

其中的值是一个 CPU mask，比如：

- `00000001` 表示只能在 CPU0 上处理
- `00000003` 表示 CPU0 和 CPU1 都可以处理

### 10.3.2_谁来真正执行这个_路由

在 ARM 平台、多数支持 GIC 的场景下，真正决定“中断要送到哪个 CPU”的是**GIC 这一层**，也就是第9章里说的“根 irq_domain 那一层”。

Linux 在你改 `smp_affinity` 时，会调用到底层 irq_chip（GIC）的 `irq_set_affinity` 回调，把这个 CPU mask 写进 GIC 的目标 CPU 寄存器。所以：

- 亲和性是 Linux 层面的接口；
- 路由是 GIC/顶层中断控制器真正执行的动作；
- 这两者必须配套，否则就会出现“我写了 affinity 但不生效”。

### 10.3.3_和_irqbalance_的关系

Linux 上常见的 irqbalance 服务会根据系统当前负载，**自动去写**这些 `/proc/irq/*/smp_affinity`，把过于集中的中断分散到别的 CPU 上。
 这就带来一个现实问题：

- 如果驱动/管理员手动绑定了某个中断到特定 CPU，但又开着 irqbalance，后者可能会把你的绑定改掉。
   解决方式通常是：
  - 对关键中断手动固定；
  - 或者在 irqbalance 配置里排除这一类中断。

### 10.3.4_为什么有的_IRQ_不能随便绑

并不是所有中断都能自由改 affinity，典型不能随便动的有：

- 本地定时器中断（per-CPU 的 PPI）；
- IPI；
- 和某些 CPU 特定资源强绑定的中断。

原因是这类中断在硬件上就是“每个 CPU 一份”，或者它的作用就是“叫醒某一颗特定 CPU”，硬件/内核会拒绝你把它绑到其他 CPU。

### 10.3.5_驱动需要做到的最小要求

对一般的外设驱动来说，最小要求只有两条：

1. **不要在驱动里硬编码中断一定要跑在 CPU0**，除非这是业务逻辑要求；
2. **如果你确实要固定中断到某个 CPU，要么在 probe 后设置，要么在文档里说明需要管理员配合关掉 irqbalance**。



## 10.4_IPI(处理器间中断)机制

### 10.4.1_IPI_是什么

IPI（Inter-Processor Interrupt）是“CPU 给另一颗 CPU 发的中断”，不是外设发的。它的目的不是“通知设备事件”，而是“让另一颗 CPU 立刻去做一件事”。在 ARM/GIC 体系里，IPI 通常对应 **SGI（Software Generated Interrupt）**，这类中断在硬件上就是“给哪颗 CPU 发就进哪颗 CPU”。

### 10.4.2_IPI_要解决的问题

- 多核调度：一个核要把任务迁到另一核，就要先把另一核叫醒或打断当前执行；
- TLB / 页表刷新：需要所有 CPU 都同步刷新时，必须用 IPI 广播；
- RCU / 时钟同步：需要对所有 CPU 做一次一致性操作时，用 IPI；
- 停机/关核：关机前要让别的核执行特定路径，也要用 IPI。

所以 IPI 属于“系统内部协调”的中断，和“外设事件”不是一类。

### 10.4.3_IPI_的特点

1. **一定是 per-CPU 的**：由哪颗 CPU 发给哪颗 CPU，是点对点或小范围广播；
2. **不会让驱动来 request_irq**：IPI 使用的是内核预留的 IRQ，驱动开发者一般不直接碰；
3. **不能随便改亲和性**：IPI 的目标 CPU 就是要被打断的那颗，改成别的 CPU 就没有意义，内核通常会拒绝；
4. **出现在 /proc/interrupts 里**：你会看到一列叫 “IPI0: …” “IPI1: …” 的计数，多核系统里这些计数变化是正常的。

### 10.4.4_IPI_和普通中断的区别

- 普通中断：外设 → 中断控制器 → 被分配到某个 CPU
- IPI：CPU → 中断控制器 → 另一颗 CPU
   也就是说，**数据流是反的**。
   所以：普通中断讲的是“怎么把一条外部事件分配给合适的 CPU”；IPI 讲的是“怎么让别的 CPU 立刻响应我”。

### 10.4.5_层级里的位置

在第9章的结构里，IPI 其实是走到了 GIC 那一层以后，直接被投递给目标 CPU 的，不走下面的 GPIO/EXTINT 这些子域。也就是：

- 外设中断：可能是 多级 → GIC → CPU
- IPI：直接 GIC → CPU

因此，**IPI 不会受你下面 GPIO irq_domain 的影响**；调试“外设中断不来”时，不要被 IPI 的计数误导。

## 10.5_中断分散策略与_irqbalance_协同

### 10.5.1_为什么要做分散

在高 PPS（网络）、高 IOPS（存储）、高频定时器这类场景下，如果所有中断都落在 CPU0：

- CPU0 的 /proc/interrupts 会高速增长，系统看起来“卡”
- 其他 CPU 空着
- 驱动的 NAPI / 下半部 也会被挤在一颗核上

所以要“分散”，即：**让不同 IRQ 落在不同 CPU 上；或让同一 IRQ 允许在一组 CPU 上运行**。

### 10.5.2_分散的三种来源

1. **手工设置**：管理员写 `/proc/irq/X/smp_affinity`；
2. **irqbalance 守护进程**：周期性评估负载，把 IRQ 调整到不同 CPU；
3. **驱动自身策略**：某些高性能驱动自己会创建多个队列/向量，然后把不同队列的中断绑到不同 CPU（典型是网卡、NVMe）。

### 10.5.3_irqbalance_的利与弊

- 利：自动化，适合“我就想均匀点”这种通用场景；
- 弊：不懂业务，可能会把你“本来要和用户线程同核”的中断挪走，导致缓存本地性变差；
- 解决：对关键 IRQ 固定亲和性，或在 irqbalance 配置中屏蔽该 IRQ。

### 10.5.4_和第9章的衔接点

第9章解决的是“这条中断最终能不能到达 GIC”；
 本章这一节解决的是“到达 GIC 后要不要换 CPU”；
 所以你在排查时应该先看：

1. 能不能到 GIC（层级对不对）
2. 到了 GIC 是不是一直在 CPU0（affinity 对不对）

不要一上来就改 affinity，否则你可能是在给一个“根本没打通的中断”配亲和性。

### 10.5.5_实际排查建议

1. 看 `/proc/interrupts`：是否所有外设 IRQ 都在同一个 CPU 上
2. 改某个 IRQ 的 affinity，看是否生效（能否跳到别的列上）
3. 如果不生效，看底层 irq_chip 是否实现了 `irq_set_affinity`（GIC 一般是有的）
4. 如果会被 irqbalance 改回去，要么停掉，要么屏蔽该 IRQ

## 10.6_驱动开发者的接口选择与注意事项

### 10.6.1_优先让系统管分配

大部分普通设备驱动，不要在代码里写死“必须在 CPU0 处理”。你只要正常地：

```c
irq = platform_get_irq(pdev, 0);
ret = devm_request_irq(dev, irq, handler, 0, dev_name(dev), dev);
```

系统就有机会通过 irqbalance / 手工 affinity 去调度。

### 10.6.2_确实要绑核的场景

只有在下面这些场景，才建议在驱动或初始化脚本里做固定：

- 这个 IRQ 要和某个特定内核线程/用户线程共享 L1/L2 缓存；
- 你在做实时/半实时控制，只能打断某颗核；
- 你要做性能对比，需要一个可重复的绑核基准。

做法一般是：

1. 驱动里 `request_irq` 成功后记录 IRQ 号；
2. 用户态脚本往 `/proc/irq/X/smp_affinity` 写固定掩码；
3. 文档注明“此驱动建议关闭 irqbalance 或屏蔽该 IRQ”。

不建议在内核驱动里直接去写 affinity 文件，因为这属于策略。

### 10.6.3_不要和层级配置打架

如果你的中断本身还在“GPIO → GIC”这种层级里，**优先把层级打通**，再谈亲和性。
 未打通就设置亲和性，只会看到“中断还是不来”，而且你还以为是 SMP 的问题，实际还是第9章的问题。

### 10.6.4_与_devm_的配合

同第9章的说明：devm 只关心“生命周期内申请的 IRQ 要不要自动释放”，不关心也不改变“这个 IRQ 被送到哪颗 CPU”。所以你完全可以：

1. devm 申请 IRQ；
2. 在用户态/脚本里调整这个 IRQ 的 affinity；
3. 设备移除时 devm 自动 free。

### 10.6.5_小结要点

- SMP 场景下的中断问题，分成两段想：先问“这条中断能到 GIC 吗”（第9章），再问“它到哪个 CPU 里去了”（本章）；
- 绝大多数中断都可以通过写 `/proc/irq/*/smp_affinity` 调整目标 CPU；
- IPI、per-CPU 定时器这类中断不能随便改；
- irqbalance 会改你手工的设置，关键中断要固定或屏蔽；
- 驱动层只要正常申请 IRQ，不要硬编码“必须 CPU0”，就能和系统的分散策略协同。
