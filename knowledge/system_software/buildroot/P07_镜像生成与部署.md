### 7.1.1_为什么镜像生成与部署很重要

到目前为止，我们已经通过 Buildroot 构建了：

- **工具链**（交叉编译环境）；
- **内核**（系统的核心）；
- **根文件系统 (rootfs)**（提供用户空间和库）；
- **包与服务**（支撑实际应用功能）。

但这些成果只是散落在 **output/** 目录下的文件：

- `output/images/zImage`（内核镜像）
- `output/images/rootfs.ext4`（文件系统镜像）
- `output/images/u-boot.bin`（引导加载程序）

要让设备真正运行起来，还需要将它们组织成 **可部署的固件镜像**，并通过合适的方式烧录或分发到目标硬件上。

这就是 **镜像生成与部署** 的任务。

------

### 7.1.2_镜像生成的目标

镜像的形式取决于：

1. **存储介质**
   - SD 卡 / eMMC → ext4、vfat
   - NAND / NOR → ubifs、jffs2
   - RAM 启动 → initramfs
2. **启动方式**
   - BootROM → SPL/TPL → U-Boot → Kernel → Rootfs
   - 是否需要分区表（MBR、GPT）
   - 是否需要引导分区和根分区
3. **产品需求**
   - 是否需要压缩（gzip、xz）
   - 是否需要签名/加密（安全启动）
   - 是否支持 OTA（远程升级）

------

### 7.1.3_部署的意义

生成镜像只是第一步，还需要部署到设备：

- **开发阶段**
  - 通过 SD 卡镜像启动。
  - 通过 NFS root 进行快速测试。
- **生产阶段**
  - 烧录到 eMMC、NAND、NOR。
  - 使用 USB/串口烧录工具。
  - OTA 或差分升级。

部署方式影响：

- 开发效率（快速测试 vs 烧录耗时）。
- 产品交付质量（可维护性、安全性）。

## 7.2_镜像类型

------

### 7.2.1_引言

在 Buildroot 中，构建过程的最终成果之一就是 **镜像文件 (image)**。
 这些镜像并不是单一形式，而是根据目标存储介质和应用场景生成多种格式，例如：

- 适合 **块设备**（SD 卡、eMMC）的 ext4；
- 适合 **闪存设备**（NAND、NOR）的 ubifs/jffs2；
- 适合 **RAM 启动**的 initramfs/cpio；
- 适合 **嵌入式只读系统**的 squashfs。

理解这些镜像的差异，有助于根据硬件环境选择合适的方案。

------

### 7.2.2_ext2/ext3/ext4

- **特征**
  - Linux 最常见的文件系统。
  - 支持目录、权限、软链接等完整 POSIX 特性。
  - ext3/ext4 支持日志功能，ext4 提供更大容量支持。
- **适用场景**
  - SD 卡、eMMC、SATA 等块存储设备。
  - 开发阶段常用 ext4 rootfs，便于调试和修改。
- **限制**
  - 直接烧录到 NAND 时效率低，不适合原始 NAND。

------

### 7.2.3_squashfs

- **特征**
  - 只读压缩文件系统，支持多种压缩算法（xz、gzip、lz4）。
  - 占用空间小，适合固件分发。
- **适用场景**
  - 路由器、机顶盒等嵌入式固件。
  - 配合 OverlayFS 提供可写层（/overlay）。
- **限制**
  - 不支持写操作，运行时修改需额外挂载 tmpfs 或 overlay。

------

### 7.2.4_initramfs_(cpio)

- **特征**
  - 打包为 cpio 格式，由内核直接解压到 RAM 中。
  - 启动速度快，不依赖外部存储。
- **适用场景**
  - 无存储介质的系统（纯 RAM 运行）。
  - 早期启动阶段（initrd/initramfs）。
  - 临时救援系统。
- **限制**
  - 占用内存，随系统运行而一直常驻 RAM。
  - 容量受限，不适合大型文件系统。

------

### 7.2.5_jffs2

- **特征**
  - Journaling Flash File System v2，针对 **NOR/NAND 原始闪存** 设计。
  - 具备写均衡和坏块管理。
- **适用场景**
  - NOR Flash（小容量）
  - 早期 NAND Flash 设备。
- **限制**
  - 大容量 NAND 上性能差，挂载时间随容量线性增加。
  - 已逐渐被 UBIFS 取代。

------

### 7.2.6_ubifs

- **特征**
  - Unsorted Block Image File System。
  - 现代 NAND Flash 专用文件系统，支持写均衡和坏块管理。
  - 比 jffs2 更高效，支持大容量。
- **适用场景**
  - 原始 NAND Flash 存储（≥ 256MB 容量）。
  - 工业控制设备、路由器。
- **限制**
  - 仅适用于原始 NAND，不适用于 eMMC/SD。

------

### 7.2.7_cramfs

- **特征**
  - 只读压缩文件系统，比 squashfs 更早出现。
  - 体积小，适合极小容量设备。
- **适用场景**
  - 资源受限的嵌入式系统。
- **限制**
  - 功能有限，逐渐被 squashfs 取代。

------

### 7.2.8_特殊镜像

- **tar**
  - 生成 tar 包格式的 rootfs，常用于开发阶段快速解包到 NFS root。
- **vfat**
  - 适合作为引导分区（boot partition），存放内核、dtb。
  - 常与 ext4 组合：
    - boot.vfat (引导分区)
    - rootfs.ext4 (根分区)
- **ubi/ubifs 镜像**
  - 用于 NAND Flash 的分区封装，结合 `ubinize` 工具生成。

------

### 7.2.9_选择策略总结

| 镜像类型  | 存储介质                   | 特点              | 典型应用             |
| --------- | -------------------------- | ----------------- | -------------------- |
| ext4      | eMMC / SD / SATA           | 全功能，读写方便  | 开发板、量产系统     |
| squashfs  | 任何存储（配合 OverlayFS） | 只读压缩          | 路由器固件           |
| initramfs | RAM                        | 快速启动          | 无存储设备，救援系统 |
| jffs2     | NOR / 小型 NAND            | 写均衡，老方案    | 小容量设备           |
| ubifs     | NAND                       | 高效写均衡        | 工控、大容量 NAND    |
| cramfs    | ROM/小闪存                 | 极简只读          | 早期嵌入式设备       |
| vfat      | 引导分区                   | U-Boot / 内核加载 | boot 分区            |
| tar       | NFS root                   | 快速测试          | 开发阶段             |

------

### 7.2.10_小结

- Buildroot 可以生成多种镜像，适配不同存储介质和应用需求。
- **块设备 → ext4**，**原始 NAND → ubifs**，**只读固件 → squashfs**，**RAM 启动 → initramfs**。
- 合理选择镜像类型，能大幅提升系统的可靠性和性能。

## 7.3_镜像生成流程

------

### 7.3.1_引言

在 Buildroot 中，镜像生成是整个构建流程的最后阶段。
 它的任务是：

- 把 **目标文件系统 (target/)** 的内容整理；
- 按照配置生成所需的镜像格式（ext4、squashfs、ubifs 等）；
- 放入统一的输出目录 `output/images/`，供烧录或部署。

要理解镜像生成流程，需要关注 Buildroot 的几个关键机制：

1. **fakeroot**（权限模拟）
2. **设备表 (device table)**
3. **mkfs 工具调用**
4. **post-image 脚本机制**

------

### 7.3.2_目标目录的准备

在前一阶段，Buildroot 会把所有软件包安装到 **`output/target/`**。

内容包括：

- 可执行文件（bin, sbin, usr/bin, usr/sbin）
- 库文件（lib, usr/lib）
- 配置文件（etc/）
- 服务脚本与 init 配置

但是此时：

- 文件所有权都是当前用户（非 root）；
- 没有特殊设备节点（如 `/dev/console`）。

因此，还需要额外处理。

------

### 7.3.3_fakeroot_权限模拟

Linux 的 rootfs 需要 root 用户权限和设备节点，例如：

- `/bin/busybox` → 需设置 `4755` 权限（setuid root）。
- `/dev/null` → 需创建字符设备节点。

在 Buildroot 中，直接在宿主机用 `root` 去做是不可取的。
 所以它引入 **fakeroot**，模拟 root 环境：

```bash
FAKEROOTDONTTRYCHOWN=1 output/host/bin/fakeroot -- ./support/scripts/fakeroot-script
```

fakeroot 的作用：

- 拦截 `chown`、`mknod` 等系统调用；
- 在文件系统镜像中记录正确的权限和设备节点；
- 不会修改宿主机文件系统。

------

### 7.3.4_设备表_(device_table)

为了让系统能正常运行，需要在 rootfs 中创建 `/dev/` 下的设备节点。

Buildroot 使用 **device_table.txt** 描述这些设备：

示例：

```
/dev/console c 622 root root 5 1
/dev/null    c 666 root root 1 3
```

含义：

- 路径
- 类型（c = 字符设备，b = 块设备）
- 权限
- 属主
- 主设备号/次设备号

在生成 rootfs 时，fakeroot 会根据该表创建节点。

------

### 7.3.5_mkfs_工具调用

不同的文件系统镜像，需要调用不同工具：

- **ext2/ext4** → `mkfs.ext4`
- **squashfs** → `mksquashfs`
- **cpio/initramfs** → `cpio`
- **jffs2** → `mkfs.jffs2`
- **ubifs** → `mkfs.ubifs + ubinize`

调用方式由 Buildroot 自动完成，用户无需手动执行。

示例（ext4）：

```bash
mkfs.ext4 -d output/target -L "rootfs" output/images/rootfs.ext4 60M
```

这里：

- `-d output/target` 指定源目录；
- `-L "rootfs"` 设置卷标；
- `60M` 来自配置 `BR2_TARGET_ROOTFS_EXT2_SIZE`。

------

### 7.3.6_输出目录结构

生成的镜像会统一存放在 **`output/images/`**：

示例：

```
output/images/
├── rootfs.ext4
├── rootfs.tar
├── rootfs.cpio
├── zImage
├── u-boot.bin
├── sdcard.img
```

- `rootfs.ext4`：ext4 根文件系统
- `rootfs.tar`：tar 打包形式
- `zImage`：内核镜像
- `sdcard.img`：完整 SD 卡镜像，包含分区表、引导分区、rootfs

------

### 7.3.7_post-image_脚本

在生成镜像后，用户可以指定 **post-image 脚本**，对产物进行进一步处理：

- 压缩镜像（gzip, xz）
- 签名/加密（安全启动）
- 生成差分更新包
- 拷贝到指定输出目录

配置路径：

```
BR2_ROOTFS_POST_IMAGE_SCRIPT
```

示例脚本 `board/myboard/post-image.sh`：

```bash
#!/bin/sh
gzip -9 $BINARIES_DIR/rootfs.ext4
cp $BINARIES_DIR/rootfs.ext4.gz /tftpboot/
```

------

### 7.3.8_镜像生成流程总结图

```
output/target/   → fakeroot 权限模拟
                 → 应用 device table
                 → 调用 mkfs 工具
                 → 生成 rootfs 镜像 (ext4/squashfs/...)
                 → post-image 脚本处理
                 → output/images/
```

------

### 7.3.9_小结

- 镜像生成的核心步骤包括 **fakeroot 权限模拟、设备表应用、mkfs 工具调用**。
- 最终所有镜像集中到 `output/images/`，便于烧录和分发。
- post-image 脚本机制提供灵活扩展，可以实现压缩、签名、定制部署。

## 7.4_镜像后处理

------

### 7.4.1_引言

在 Buildroot 完成 rootfs 和镜像的生成后，产物通常存放在 `output/images/`。
 然而，这些“原始镜像”并不总是能直接交付或用于量产。

- **存储和分发**：需要压缩，节省带宽和存储空间。
- **安全性**：需要签名或加密，以防止固件被篡改。
- **产品化**：需要对镜像进行后处理，添加版本信息、差分更新支持等。

Buildroot 提供了 **镜像后处理机制 (post-image)**，允许开发者在镜像生成之后添加自定义步骤。

------

### 7.4.2_压缩

常见压缩方式：

- **gzip**

  - 压缩速度快，支持广泛。

  - 适合量产分发。

  - 命令：

    ```bash
    gzip -9 rootfs.ext4
    ```

- **xz**

  - 压缩率更高，但速度较慢。

  - 适合对存储空间要求极高的系统。

  - 命令：

    ```bash
    xz -9 rootfs.ext4
    ```

- **lz4**

  - 压缩率低，但解压极快。

  - 常用于快速启动的嵌入式设备。

  - 命令：

    ```bash
    lz4 rootfs.ext4
    ```

压缩后的文件通常命名为：

```
rootfs.ext4.gz
rootfs.ext4.xz
rootfs.ext4.lz4
```

------

### 7.4.3_签名与校验

为了保证固件的完整性与安全性，可以在后处理阶段进行签名。

- **哈希校验**

  ```bash
  sha256sum rootfs.ext4 > rootfs.ext4.sha256
  ```

- **GPG 签名**

  ```bash
  gpg --output rootfs.ext4.sig --detach-sig rootfs.ext4
  ```

- **安全启动 (Secure Boot)**

  - 在部分平台（如 i.MX、TI AM335x）需要 U-Boot 或内核签名。
  - Buildroot 可以在 post-image 脚本中调用厂商工具（如 `mkimage`、`sign_tool`）。

------

### 7.4.4_post-image_脚本机制

在 `menuconfig` 中配置：

```
System configuration  --->
    (path/to/post-image.sh) Root filesystem post-image script
```

该脚本会在所有镜像生成之后执行，变量环境包括：

- **$BINARIES_DIR**：镜像输出目录（`output/images/`）
- **$TARGET_DIR**：目标 rootfs 目录（`output/target/`）

示例 `board/myboard/post-image.sh`：

```bash
#!/bin/sh
# 压缩 rootfs
gzip -9 $BINARIES_DIR/rootfs.ext4

# 生成校验文件
sha256sum $BINARIES_DIR/rootfs.ext4.gz > $BINARIES_DIR/rootfs.ext4.gz.sha256

# 拷贝到 TFTP 目录，供测试板启动
cp $BINARIES_DIR/rootfs.ext4.gz /tftpboot/myboard/
```

------

### 7.4.5_典型后处理任务

- **压缩镜像**（gzip, xz, lz4）
- **生成校验和**（md5, sha256）
- **添加签名**（gpg, secure boot）
- **拷贝镜像**到指定目录（如 `/tftpboot`，供网络启动）
- **构建 OTA 包**
  - 将 rootfs 和版本信息打包为 `.zip` 或 `.swu`
  - 用于差分升级或远程更新

------

### 7.4.6_使用场景对比

| 后处理操作             | 目的                   | 使用场景           |
| ---------------------- | ---------------------- | ------------------ |
| 压缩 (gzip/xz/lz4)     | 节省存储空间、加快下载 | 固件分发、OTA      |
| 校验 (sha256)          | 确保完整性             | 烧录验证、OTA 校验 |
| 签名 (gpg/secure boot) | 防篡改                 | 安全产品、车载系统 |
| OTA 包构建             | 远程升级               | 智能硬件、IoT      |

------

### 7.4.7_小结

- Buildroot 的后处理机制允许开发者在镜像生成之后灵活定制。
- 常见操作包括 **压缩、校验、签名、拷贝、打包**。
- 通过 `post-image.sh` 脚本，可以把 Buildroot 的基础构建结果转化为产品级的交付固件。

## 7.5_部署方式

------

### 7.5.1_引言

镜像生成只是软件构建的一部分，最终目标是让 **硬件设备能够运行这些镜像**。
 部署方式因 **硬件存储介质**、**启动链** 和 **开发/量产需求** 不同而有所差异。

Buildroot 并不直接负责部署，但它生成的产物正好适配各种常见的部署流程。

------

### 7.5.2_SD_卡/TF_卡烧录

#### (1)_步骤

1. 在 Buildroot 中选择 SD 卡镜像生成：

   - `BR2_TARGET_ROOTFS_EXT2` 或 `BR2_TARGET_ROOTFS_EXT4`
   - `BR2_TARGET_ROOTFS_TAR`（配合 `genimage` 生成完整分区表镜像）

2. 输出镜像：

   ```
   output/images/sdcard.img
   ```

3. 烧录到 SD 卡：

   ```bash
   sudo dd if=output/images/sdcard.img of=/dev/sdX bs=4M status=progress
   sync
   ```

#### (2)_特点

- 开发阶段最常用。
- 可反复写入，快速测试。

------

### 7.5.3_eMMC/NAND_烧录

#### (1)_eMMC

- 类似 SD 卡，可以直接 `dd` 烧录。
- 通常分为 **boot 分区** 和 **rootfs 分区**：
  - boot 分区：存放 U-Boot、内核、设备树。
  - rootfs 分区：存放 rootfs 镜像。

#### (2)_NAND

- 需要使用 `flashcp`、`nandwrite` 等工具。

- 文件系统需选择 jffs2 或 ubifs。

- 通常还要配合 **ubinize** 工具生成 UBI 镜像：

  ```bash
  ubinize -o rootfs.ubi -m 2048 -p 128KiB -s 512 ubinize.cfg
  ```

#### (3)_特点

- eMMC：适合量产，稳定可靠。
- NAND：适合大容量存储，但部署更复杂。

------

### 7.5.4_网络启动

#### (1)_TFTP_+_NFS

- **内核镜像** 通过 TFTP 下载：

  - U-Boot 命令：

    ```bash
    tftp 0x82000000 zImage
    tftp 0x83000000 imx6ull-evk.dtb
    bootz 0x82000000 - 0x83000000
    ```

- **rootfs** 挂载在 NFS：

  - 内核启动参数：

    ```
    root=/dev/nfs nfsroot=192.168.1.100:/nfsroot,v3,tcp ip=dhcp
    ```

#### (2)_特点

- 适合开发调试，避免频繁烧录。
- 依赖稳定的网络环境。

------

### 7.5.5_USB/串口烧录工具

部分 SoC 提供专用烧录工具：

- NXP i.MX → `uuu`（Universal Update Utility）
- Allwinner → `PhoenixSuit`
- Rockchip → `rkdeveloptool`
- STM32MP1 → `STM32CubeProgrammer`

这些工具通常通过 **USB OTG** 或 **UART** 把镜像直接写入 eMMC/NAND。

特点：

- 常用于量产阶段。
- 可与 Buildroot 的 post-image 脚本结合，实现自动拷贝。

------

### 7.5.6_OTA_升级

对于远程设备，手动烧录不可行，需要 OTA (Over-The-Air) 升级机制。

#### (1)_实现方式

1. **完整固件替换**
   - 下载新的 rootfs 镜像，覆盖旧分区。
   - 简单但耗时，带宽消耗大。
2. **差分升级**
   - 使用工具生成补丁（如 `bsdiff`）。
   - 只更新变更部分，节省带宽。
3. **A/B 分区机制**
   - 两个 rootfs 分区，升级时写入备用分区。
   - 升级失败可回滚，提高可靠性。

#### (2)_特点

- 需要在 Buildroot 中集成 OTA 相关软件包（如 `swupdate`、`rauc`）。
- post-image 脚本可用来打包 OTA 镜像（zip/swu）。

------

### 7.5.7_部署方式对比表

| 部署方式     | 特点                 | 适用阶段        |
| ------------ | -------------------- | --------------- |
| SD 卡烧录    | 简单快速，可重复写入 | 开发阶段        |
| eMMC 烧录    | 稳定可靠，适合量产   | 生产阶段        |
| NAND 烧录    | 支持大容量，但复杂   | 工业/大容量存储 |
| 网络启动     | 快速测试，无需烧录   | 开发调试        |
| USB/串口烧录 | 厂商工具支持，可靠   | 量产烧录        |
| OTA 升级     | 远程更新，支持回滚   | 产品交付与维护  |

------

### 7.5.8_小结

- 部署方式取决于 **硬件介质**（SD 卡、eMMC、NAND）、**使用场景**（开发、量产、远程升级）。
- 开发阶段推荐 **SD 卡/网络启动**；
- 量产阶段推荐 **eMMC 烧录或 USB 工具**；
- 远程设备推荐 **OTA 升级**。

## 7.6_调试与验证

------

### 7.6.1_引言

生成镜像并烧录到设备只是第一步。
 在实际开发和量产之前，必须确保镜像内容完整、能够正确启动，并能提供所需的功能。

Buildroot 提供了多种调试与验证方法，从 **宿主机检查** 到 **目标设备运行测试**，帮助开发者快速发现并修复问题。

------

### 7.6.2_在宿主机检查镜像

#### (1)_挂载镜像内容

```bash
sudo mount -o loop output/images/rootfs.ext4 /mnt
ls /mnt/bin
```

- 可以直接查看文件是否存在（如 `/bin/busybox`）。
- 确认 `/etc/inittab`、`/etc/passwd` 等配置文件。

#### (2)_解包_tar/cpio_镜像

```bash
tar -tf output/images/rootfs.tar | less
cpio -it < output/images/rootfs.cpio | less
```

- 用于检查 rootfs 目录结构。

------

### 7.6.3_使用_QEMU_测试镜像

Buildroot 提供了 QEMU 支持，可以不依赖目标板先验证系统。

示例（ARM）：

```bash
qemu-system-arm -M vexpress-a9 -kernel output/images/zImage \
    -dtb output/images/vexpress-v2p-ca9.dtb \
    -drive file=output/images/rootfs.ext4,if=sd,format=raw \
    -append "root=/dev/mmcblk0 rw console=ttyAMA0"
```

优势：

- 无需烧录，快速验证系统启动流程。
- 可调试内核日志、文件系统挂载情况。

------

### 7.6.4_在目标设备验证

#### (1)_串口日志

- 连接串口，观察 U-Boot → Kernel → Init 的启动过程。
- 确认是否挂载 rootfs、是否进入 shell。

#### (2)_服务验证

- 检查服务是否随系统启动：
  - BusyBox init → `/etc/inittab`
  - SysV init → `/etc/init.d/`
  - systemd → `systemctl status`

#### (3)_程序运行测试

- 在设备上执行：

  ```bash
  /usr/bin/hello
  ```

- 确认自定义包或应用已正确安装。

------

### 7.6.5_动态库依赖检查

在设备上：

```bash
ldd /usr/bin/hello
```

- 确认所有依赖库都已安装。
- 如果提示 `not found`，需要检查 Buildroot 包依赖声明。

在宿主机交叉工具链环境：

```bash
output/host/bin/arm-linux-readelf -d output/target/usr/bin/hello | grep NEEDED
```

- 可提前发现依赖缺失问题。

------

### 7.6.6_调试工具

- **strace**
   跟踪系统调用：

  ```bash
  strace /usr/bin/hello
  ```

- **gdb / gdbserver**
   在目标上运行 gdbserver，在宿主机用交叉 gdb 远程调试。

- **dmesg**
   查看内核日志，检查驱动加载、设备挂载问题。

- **journalctl**（systemd）
   查看服务日志：

  ```bash
  journalctl -u myservice
  ```

------

### 7.6.7_常见问题与排查

| 问题                      | 可能原因                 | 解决方法                             |
| ------------------------- | ------------------------ | ------------------------------------ |
| 系统卡在启动阶段          | rootfs 没有正确挂载      | 检查内核启动参数，确认 root=/dev/... |
| 提示 `/bin/sh: not found` | BusyBox 未安装或路径错误 | 启用 `BR2_PACKAGE_BUSYBOX`，重新构建 |
| 服务未启动                | init 脚本错误或缺少依赖  | 检查 inittab/init.d/.service 配置    |
| 应用缺少库                | 包依赖未声明             | 修改 `.mk`，添加 `XXX_DEPENDENCIES`  |
| 启动后闪退                | 二进制架构不符           | 检查交叉工具链架构配置               |

------

### 7.6.8_验证流程总结

```
宿主机挂载/解包镜像
   ↓
QEMU 启动测试（可选）
   ↓
目标设备串口日志验证
   ↓
检查 rootfs 与服务配置
   ↓
运行应用，确认依赖完整
   ↓
使用 strace/gdb/dmesg 进行问题排查
```

------

### 7.6.9_小结

- **宿主机验证**：挂载/解包 rootfs，检查文件和配置。
- **QEMU 验证**：无需目标硬件即可测试启动流程。
- **目标设备验证**：串口日志、服务状态、库依赖检查。
- **调试工具**：strace、gdb、dmesg、journalctl 帮助快速定位问题。

通过这些方法，可以确保 Buildroot 生成的镜像不仅能启动，还能稳定运行并满足应用需求。

## 7.7_小结

------

### 7.7.1_镜像生成与部署的核心作用

镜像是 Buildroot 构建过程的 **最终交付物**。
 它将内核、根文件系统和必要的启动文件打包，转化为能够在目标硬件上运行的固件。

没有镜像，前面所有的编译与集成都只能停留在宿主机；
 有了镜像，系统才能真正进入嵌入式设备的世界。

------

### 7.7.2_本章回顾

1. **引言 (7.1)**
   - 镜像生成连接了构建产物与目标硬件。
   - 部署方式因存储介质、启动链和产品需求而异。
2. **镜像类型 (7.2)**
   - ext4 → 适合 SD 卡/eMMC；
   - squashfs → 适合只读固件；
   - initramfs → 适合无存储设备或早期启动；
   - jffs2/ubifs → 针对原始 NAND；
   - 还包括 cramfs、tar、vfat 等特殊用途格式。
3. **镜像生成流程 (7.3)**
   - 基于 `output/target` 的内容。
   - 使用 **fakeroot** 模拟 root 权限。
   - 根据 **device table** 创建节点与权限。
   - 调用 mkfs 工具生成 ext4/squashfs/ubifs 等镜像。
   - 最终产物集中在 `output/images/`。
4. **镜像后处理 (7.4)**
   - 常见任务：压缩、签名、校验、拷贝。
   - 通过 post-image 脚本灵活扩展。
   - 支持 OTA 升级包构建。
5. **部署方式 (7.5)**
   - SD 卡烧录：开发调试最常用。
   - eMMC 烧录：量产阶段首选。
   - NAND 烧录：需特殊工具，适合大容量闪存。
   - 网络启动 (TFTP + NFS)：快速测试。
   - USB/串口烧录工具：厂商提供，常用于量产。
   - OTA 升级：远程维护设备的核心机制。
6. **调试与验证 (7.6)**
   - 宿主机检查：挂载 rootfs、解包镜像。
   - QEMU 测试：不依赖硬件即可验证启动。
   - 目标板调试：串口日志、服务状态、依赖检查。
   - 工具：strace、gdb、dmesg、journalctl。

------

### 7.7.3_设计哲学总结

- **轻量化**：Buildroot 不负责部署工具，只负责生成产物。
- **灵活性**：通过 post-image 脚本和 overlay，可以高度定制。
- **专注性**：目标是生成确定性的、可交付的固件，而不是运行时包管理。

------

### 7.7.4_对比与定位

| 阶段     | Buildroot                    | Yocto                   |
| -------- | ---------------------------- | ----------------------- |
| 镜像生成 | 内置 mkfs 脚本，简单直观     | recipe 更复杂，灵活度高 |
| 部署支持 | 提供产物，不负责烧录         | meta 层可集成厂商工具   |
| OTA 升级 | 需外部软件（swupdate、rauc） | meta 层已有成熟支持     |

Buildroot 的定位非常明确：

- **快速生成确定性的固件镜像**；
- 部署、升级等环节由开发者或外部工具完成。
