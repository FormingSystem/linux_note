# 第11章_调试与问题排查

## 11.1_引言

在前十部分，我们已经完整学习了 Buildroot 的 **构建流程**：

- 工具链生成；
- 内核与 Bootloader 的编译；
- Rootfs 与包管理；
- 镜像生成与部署。

从理论上说，完成这些步骤后，我们应该能得到一个可以直接启动并运行的嵌入式系统。
 但在实践中，**问题和错误几乎不可避免**：

- 构建阶段：
  - 某个包下载失败；
  - 交叉编译器版本不兼容；
  - 补丁应用出错。
- 部署阶段：
  - 镜像无法启动；
  - U-Boot 找不到内核；
  - Rootfs 无法挂载。
- 运行阶段：
  - Kernel panic；
  - 程序缺少共享库；
  - 权限、依赖问题导致应用运行失败。

这些问题往往跨越多个层次：**构建系统 → 启动链路 → 用户空间**，
 因此需要系统化的调试与排查思路。

## 11.2_构建阶段的常见问题与调试

------

### 11.2.1_背景

Buildroot 的第一大难关往往出现在 **构建阶段**。
 此时问题主要集中在：

- **下载失败**（源站点不可用、代理问题）；
- **依赖缺失**（宿主机少装了工具/库）；
- **编译错误**（GCC 版本不匹配、补丁应用失败）；
- **配置错误**（包选项冲突）。

这些问题虽然常见，但排查起来可能比较繁琐，
 因此需要掌握 **定位 → 清理 → 重试** 的基本方法。

------

### 11.2.2_下载失败

#### (1)_常见现象

- `wget: unable to resolve host address`
- `sha256sum: mismatch`
- `Package xyz not found`

#### (2)_排查方法

1. 检查网络与代理：

   - Buildroot 默认不使用宿主机代理 → 需要在 `BR2_PRIMARY_SITE` 中指定本地镜像源。
   - 或者在 `make menuconfig` → `Toolchain` → 启用下载代理。

2. 验证源是否可用：

   ```bash
   wget http://example.com/package.tar.gz
   ```

3. 使用本地缓存：

   - 把压缩包放入 `dl/` 目录（默认存放下载文件）。
   - Buildroot 会直接使用本地文件，而不是重新下载。

------

### 11.2.3_依赖缺失

#### (1)_常见现象

- `autoconf: command not found`
- `flex: not found`
- `python3: module xxx not found`

#### (2)_排查方法

1. 确认宿主机环境：
   - Debian/Ubuntu: `sudo apt-get install build-essential git bison flex gettext`
   - Fedora: `sudo dnf groupinstall "Development Tools"`
2. 使用 Buildroot 文档提供的依赖列表：
   - `docs/manual/prerequisites.txt`
   - 或者执行 `make` 时，Buildroot 会提示缺少的包。

------

### 11.2.4_编译错误

#### (1)_常见现象

- `error: conflicting types for 'foo'`
- `undefined reference to 'bar'`
- `gcc: internal compiler error`

#### (2)_排查方法

1. 确认工具链版本

   - 部分包对 GCC 有版本要求（例如 U-Boot 文档 `doc/build/gcc.rst` 会注明最低/最高版本）。
   - 可以在 Buildroot 中选择内置工具链（internal toolchain），避免依赖宿主机版本。

2. 清理并重试

   - 只清理失败的包：

     ```bash
     make <pkg>-dirclean
     make <pkg>
     ```

   - 清理整个目标：

     ```bash
     make clean
     make
     ```

3. 检查补丁是否正确

   - 补丁失败常见于上游版本更新。
   - 查看 `output/build/<pkg>/patch.log`，确认是否能手动应用。

------

### 11.2.5_配置错误

#### (1)_常见现象

- 两个库冲突，导致编译失败；
- Rootfs 过大，超出分区限制。

#### (2)_排查方法

1. 使用 `make menuconfig` 检查包配置：
   - 避免同时启用同类功能的多个库（如 openssl vs mbedtls）。
2. 检查文件系统大小：
   - 在 `Filesystem images` 菜单调整 `BR2_TARGET_ROOTFS_EXT2_SIZE`。
   - 根据 log 中的提示扩容。

------

### 11.2.6_常用调试技巧

- **只构建单个包**

  ```bash
  make <pkg>
  ```

  例如 `make busybox`。

- **查看命令执行过程**

  ```bash
  make V=1
  ```

  显示完整 gcc/ld 命令行。

- **模拟执行（dry-run）**

  ```bash
  make -n
  ```

  只显示将要执行的命令，不真正执行。

- **调试脚本**
   进入 `output/build/<pkg>/`，手动执行 `configure`/`make`，确认失败点。

------

### 11.2.7_小结

- 构建阶段的问题大多数来自 **下载、依赖、编译、配置**；
- 关键是掌握 **局部清理 (`<pkg>-dirclean`) → 重试 → 分析 log** 的流程；
- 工具链版本与补丁兼容性是高频问题，需要特别关注。

## 11.3_镜像生成与部署问题

------

### 11.3.1_背景

在 **构建阶段** 成功生成了内核、rootfs 和 bootloader 后，Buildroot 会进入 **镜像生成阶段**：

- genimage 组合分区镜像（如 `sdcard.img`）；
- post-image 脚本执行额外任务；
- 最终输出可烧录的镜像文件。

问题可能出现在：

1. **镜像生成失败**（配置错误、空间不足）；
2. **镜像烧录错误**（`dd` 操作失败、写入错盘）；
3. **设备无法启动**（分区偏移错误、Bootloader 未找到内核）。

------

### 11.3.2_genimage_配置错误

#### (1)_常见现象

- `genimage: error: unknown option`
- `partition rootfs: file rootfs.ext4 not found`
- `boot.vfat: No space left on device`

#### (2)_排查方法

1. 检查 `genimage.cfg`：

   - 确认 `image = "rootfs.ext4"` 文件是否存在于 `output/images/`；
   - 确认分区大小足够。

2. 扩大 rootfs 分区：

   - 修改 Buildroot 配置：

     ```
     Filesystem images → ext2/3/4 root filesystem → exact size
     ```

   - 或者在 `genimage.cfg` 中调整 `size`。

3. 调试 genimage：

   ```bash
   genimage --loglevel 2 ...
   ```

   以查看更多错误信息。

------

### 11.3.3_分区偏移错误

#### (1)_常见现象

- 烧录后设备无法启动；
- U-Boot 输出 `Bad MBR partition` 或找不到内核。

#### (2)_原因

- SoC 要求 Bootloader 在特定偏移，例如：
  - NXP i.MX 系列：通常偏移 **1KB**；
  - Rockchip SoC：Bootloader 需要写在 **固定 sector**。

#### (3)_解决办法

- 检查 `genimage.cfg` 的 `offset` 配置是否正确。
- 参考厂商手册，确保 Bootloader 被放置在 SoC 启动 ROM 可识别的位置。

------

### 11.3.4_烧录失败

#### (1)_常见现象

- `dd: failed to open '/dev/sdX': Permission denied`
- 写入成功，但板子启动不了。

#### (2)_排查方法

1. 权限问题：

   - 使用 `sudo dd`。

2. 确认目标设备：

   - `lsblk` 或 `dmesg | tail` 查看 SD 卡挂载路径；
   - 避免误操作覆盖 PC 的系统盘。

3. 校验写入结果：

   ```bash
   sudo cmp output/images/sdcard.img /dev/sdX
   ```

   确保镜像写入无误。

------

### 11.3.5_设备启动失败

#### (1)_常见现象

- U-Boot 启动后提示 `No kernel image found`；
- Kernel panic，找不到 rootfs。

#### (2)_排查方法

1. 检查 Boot 分区内容：
   - 是否包含 `zImage` 和 `.dtb`；
   - 是否和 U-Boot 的 `bootcmd` 配置一致。
2. 检查 rootfs 类型：
   - 如果 rootfs 使用 SquashFS，需确保内核支持 `CONFIG_SQUASHFS`；
   - 如果是 UBIFS，需保证启用了 `CONFIG_MTD_UBI`。
3. 串口日志分析：
   - 观察 Bootloader 输出的内核加载地址；
   - 确认与内核编译时指定的 `LOADADDR` 是否一致。

------

### 11.3.6_常见调试手段

- **解包镜像**

  - `fdisk -l sdcard.img` 查看分区布局；
  - `mount -o loop,offset=<X> sdcard.img /mnt` 挂载验证分区内容。

- **检查镜像内容**

  - `hexdump -C sdcard.img | head` 确认 Bootloader 是否写入正确偏移。

- **模拟启动**

  - 使用 QEMU 启动镜像，验证能否挂载 rootfs：

    ```bash
    qemu-system-arm -M vexpress-a9 -kernel zImage -dtb vexpress.dtb -sd sdcard.img
    ```

------

### 11.3.7_小结

- 镜像生成问题多由 **genimage.cfg 配置错误** 和 **rootfs 大小不足** 引起；
- 分区偏移问题与 **SoC 启动要求** 强相关，需要查阅厂商手册；
- 烧录失败常见于 **写入权限或目标设备选择错误**；
- 启动失败则需结合 **串口日志 + rootfs 配置**来定位。

## 11.4_启动阶段问题排查

------

### 11.4.1_背景

镜像烧录完成后，设备进入 **启动阶段**：

1. **BootROM → SPL/TPL → U-Boot**
2. **U-Boot → Linux Kernel**
3. **Linux Kernel → Rootfs**

任何一个环节出错，都可能导致设备无法进入用户空间。
 因此，调试启动问题需要分层排查，从 **最底层 Bootloader** 到 **内核挂载 rootfs**。

------

### 11.4.2_U-Boot_启动失败

#### (1)_常见现象

- 串口无输出 → SPL/U-Boot 未正确写入偏移；
- 输出乱码 → 串口波特率配置错误；
- 报错 `Bad MBR partition` 或 `Unrecognized filesystem type`。

#### (2)_排查方法

1. **确认 Bootloader 写入位置**

   - i.MX 系列通常偏移 1KB；
   - Rockchip、Allwinner 等有固定 sector 要求。

2. **检查编译配置**

   - 确认使用的 defconfig 与目标板一致：

     ```bash
     make mx6ull_14x14_evk_defconfig
     ```

3. **串口调试**

   - 使用 `minicom` / `picocom`，波特率（常见 115200）要与 U-Boot 配置匹配。

------

### 11.4.3_Kernel_加载失败

#### (1)_常见现象

- `Unrecognized image format`
- `Wrong Image Format for bootm command`
- `Error: bad magic number`

#### (2)_排查方法

1. **确认内核格式**

   - 如果 U-Boot 使用 `bootm`，需要 `uImage`（mkimage 生成）；
   - 如果使用 `bootz`，则需要 `zImage`。

2. **检查设备树 (.dtb)**

   - 确认内核启动参数中指定了正确的 `.dtb` 文件；
   - `fdt addr` 命令确保 dtb 地址正确。

3. **检查加载地址**

   - U-Boot 的 `loadaddr` 与内核编译时 `CONFIG_LOADADDR` 必须一致。

   - 使用命令：

     ```bash
     printenv bootcmd
     printenv bootargs
     ```

------

### 11.4.4_Kernel_panic

#### (1)_常见现象

- `Kernel panic - not syncing: VFS: Unable to mount root fs on unknown-block(0,0)`
- `No init found. Try passing init= option to kernel.`

#### (2)_排查方法

1. **检查 rootfs 镜像类型**

   - 内核必须支持对应的文件系统：
     - ext4 → `CONFIG_EXT4_FS=y`
     - SquashFS → `CONFIG_SQUASHFS=y`
     - UBIFS → `CONFIG_UBIFS_FS=y`

2. **检查 root 参数**

   - 例如：

     ```bash
     root=/dev/mmcblk0p2 rootwait rw
     ```

   - 确认分区号正确。

3. **确认 rootfs 是否写入**

   - 在宿主机挂载 sdcard.img 检查 rootfs 分区内容。

------

### 11.4.5_Rootfs_挂载失败

#### (1)_常见现象

- `mount: No such file or directory`
- `EXT4-fs (mmcblk0p2): bad geometry: block count too large`

#### (2)_排查方法

1. **检查内核命令行 bootargs**

   - `root=/dev/mmcblk0p2` 是否和分区表匹配？
   - NAND/UBIFS 情况下，需指定 `ubi.mtd=... root=ubi0:rootfs`。

2. **验证文件系统一致性**

   - 在宿主机挂载 rootfs 分区：

     ```bash
     sudo mount -o loop,offset=... sdcard.img /mnt
     ```

3. **确认 init 程序存在**

   - `output/target/sbin/init` 或 `/bin/sh` 必须存在。
   - 如果缺少，可以在 `System configuration` → `Init system` 中选择 BusyBox init/systemd。

------

### 11.4.6_串口日志调试

串口输出是启动阶段调试的核心工具：

- **无输出** → Bootloader 未写入 / 偏移错误。
- **输出停在 U-Boot** → Bootcmd 或内核镜像错误。
- **输出停在 Kernel** → 文件系统挂载失败。
- **输出进入 Shell** → 启动成功。

建议在 post-image 阶段保存完整的串口 log，以便后续对比分析。

------

### 11.4.7_小结

- **U-Boot 阶段**：重点检查 Bootloader 写入位置、串口波特率、内核格式（zImage/uImage）。
- **内核阶段**：检查 dtb 文件、加载地址、bootargs 设置。
- **rootfs 阶段**：检查文件系统支持、root 参数、init 程序存在性。
- **调试工具**：串口日志是关键，必要时配合 QEMU 模拟验证。

## 11.5_用户空间问题排查

------

### 11.5.1_背景

当系统顺利完成 **Bootloader → 内核 → rootfs 挂载**，就进入了 **用户空间**。
 在这个阶段，内核已经启动，但可能会遇到：

- **init 进程错误**（没有 init 或 init 配置错误）；
- **应用无法运行**（缺少库或依赖）；
- **命令找不到**（BusyBox 配置过于精简）；
- **动态库路径问题**（ld.so 配置缺失）。

这些问题比启动阶段更隐蔽，需要仔细检查 rootfs 的完整性和运行环境。

------

### 11.5.2_init_相关问题

#### (1)_常见现象

- `Kernel panic - not syncing: No init found`
- 系统停在空白界面，无法交互。

#### (2)_排查方法

1. 检查 rootfs 中是否有 `/sbin/init`：

   - Buildroot 默认使用 **BusyBox init**；
   - 可以在 `System configuration → Init system` 中选择：
     - BusyBox init
     - systemd
     - None（仅适合极简系统）。

2. 内核参数指定 init：

   - 在 bootargs 中添加：

     ```bash
     init=/bin/sh
     ```

   - 便于调试 rootfs 是否可运行。

------

### 11.5.3_BusyBox_配置缺失

#### (1)_常见现象

- `ls: not found`
- `ifconfig: command not found`

#### (2)_原因

BusyBox 是高度可裁剪的，部分命令默认未启用。

#### (3)_排查方法

1. 进入 Buildroot 菜单：

   ```bash
   make busybox-menuconfig
   ```

   启用需要的命令。

2. 重新构建 BusyBox：

   ```bash
   make busybox-dirclean
   make busybox
   ```

3. 验证命令是否已安装在 `/bin` 或 `/sbin`。

------

### 11.5.4_动态库缺失

#### (1)_常见现象

- `error while loading shared libraries: libxyz.so: cannot open shared object file: No such file or directory`

#### (2)_排查方法

1. 检查库是否存在：

   ```bash
   ls output/target/lib | grep libxyz
   ```

2. 确认包是否启用了对应依赖：

   - 例如 `openssl` vs `mbedtls`，某些程序需要特定库。

3. 使用 `ldd` 查看依赖：

   ```bash
   ldd myapp
   ```

4. 配置动态库搜索路径：

   - 编辑 `/etc/ld.so.conf` 或 `/etc/ld.so.cache`（Buildroot 默认可能没有）。

   - 或者设置环境变量：

     ```bash
     export LD_LIBRARY_PATH=/usr/lib
     ```

------

### 11.5.5_配置文件缺失

#### (1)_常见现象

- `Error: failed to open /etc/resolv.conf`
- `hostname: command not found`

#### (2)_排查方法

1. 检查 Buildroot 是否启用了相应包：
   - 例如 `resolv.conf` 来自 `busybox` 的网络配置；
   - `hostname` 来自 `inetutils` 或 `util-linux`。
2. 在 `output/target/etc/` 中确认配置文件是否存在。
3. 重新构建 rootfs 镜像。

------

### 11.5.6_权限与设备文件

#### (1)_常见现象

- `Permission denied`
- `cannot open /dev/ttyS0: No such file or directory`

#### (2)_排查方法

1. 确认 `/dev` 下设备节点是否存在：

   - Buildroot 默认通过 `makedevs` 或 `udev` 创建。

2. 检查权限：

   ```bash
   ls -l /dev/ttyS0
   ```

3. 如果缺失，可在 Buildroot 的 `System configuration → /dev management` 中选择：

   - None
   - devtmpfs
   - BusyBox mdev
   - eudev

------

### 11.5.7_调试工具

在用户空间调试时，可以借助以下工具：

- `strace` → 跟踪系统调用，找出失败原因。
- `ldd` → 检查二进制依赖库。
- `hexdump` / `xxd` → 检查二进制文件是否损坏。
- `mount` / `df` → 验证文件系统是否挂载正确。

在 Buildroot 中，可以通过选中 `strace` 包来启用。

------

### 11.5.8_小结

- 用户空间问题主要集中在 **init、BusyBox、库依赖、配置文件、设备文件**。
- 关键调试手段：
  - **init=/bin/sh** 启动 rootfs；
  - **ldd/strace** 定位依赖问题；
  - **busybox-menuconfig** 重新裁剪命令集。
- 一般思路：确认 rootfs 完整性 → 检查库与配置 → 验证 init 系统。

## 11.6_调试工具与方法

------

### 11.6.1_背景

前面几节我们已经覆盖了 **构建 → 镜像生成 → 启动 → 用户空间** 各阶段的问题。
 无论在哪个环节出现问题，**高效的调试工具**都是定位和解决问题的关键。

在嵌入式系统中，常见的调试工具主要包括：

- **串口调试输出**：最基本的日志来源；
- **系统调用跟踪 (`strace`)**：排查应用执行异常；
- **动态库依赖 (`ldd`)**：验证二进制依赖完整性；
- **模拟器 (`QEMU`)**：无需实际硬件即可测试镜像；
- **日志系统 (`dmesg`, `/var/log/`)**：查看内核和用户空间错误信息。

------

### 11.6.2_串口调试

#### (1)_原理

- 几乎所有 SoC 都支持 **UART 调试接口**；
- 从 BootROM 开始，Bootloader 和内核都会输出日志到串口。

#### (2)_使用方法

1. 确认串口线连接正确；

2. 在宿主机上使用工具：

   ```bash
   picocom -b 115200 /dev/ttyUSB0
   ```

   或者：

   ```bash
   minicom -D /dev/ttyUSB0 -b 115200
   ```

#### (3)_调试要点

- **无输出**：Bootloader 没有启动或写入错误位置；
- **乱码**：波特率不匹配；
- **停在 U-Boot**：检查 `bootcmd` 与内核镜像是否匹配；
- **停在 Kernel**：检查 rootfs 参数。

------

### 11.6.3_strace

#### (1)_作用

- 跟踪用户程序执行的 **系统调用**，发现文件/库缺失、权限错误。

#### (2)_用法

```bash
strace ./myapp
```

#### (3)_示例输出

```
open("/etc/config", O_RDONLY) = -1 ENOENT (No such file or directory)
```

说明程序运行失败是因为缺少配置文件。

在 Buildroot 中，可以在 `menuconfig` 里启用 `strace` 包。

------

### 11.6.4_ldd

#### (1)_作用

- 显示可执行文件依赖的动态库。

#### (2)_用法

```bash
ldd ./myapp
```

#### (3)_示例输出

```
libm.so.6 => not found
```

说明 `libm` 没有包含在 rootfs 中，需要在 Buildroot 配置里启用对应库。

------

### 11.6.5_QEMU_模拟

#### (1)_优势

- 不需要真实硬件即可快速测试镜像；
- 适合验证 rootfs 是否可启动。

#### (2)_示例命令

```bash
qemu-system-arm -M vexpress-a9 \
  -kernel zImage \
  -dtb vexpress-v2p-ca9.dtb \
  -sd output/images/rootfs.ext4 \
  -append "root=/dev/mmcblk0 rw console=ttyAMA0"
```

#### (3)_应用场景

- 在硬件尚未到位时提前开发；
- 调试内核和 rootfs 是否匹配；
- 验证 init 系统和 BusyBox 是否正常工作。

------

### 11.6.6_日志工具

1. **内核日志**
   - `dmesg` → 查看启动后内核输出；
   - 常见于驱动加载失败、rootfs 挂载异常。
2. **用户空间日志**
   - `/var/log/messages`、`/var/log/syslog`；
   - BusyBox init 默认可能不生成，需要额外配置。
3. **自定义调试输出**
   - 在 init 脚本或应用程序中添加 `echo` 输出到串口。

------

### 11.6.7_常见调试策略

| 场景            | 工具         | 调试思路                         |
| --------------- | ------------ | -------------------------------- |
| 系统卡在 U-Boot | 串口         | 检查 Bootloader 偏移与 `bootcmd` |
| Kernel panic    | 串口 + dmesg | 确认 rootfs 类型与内核配置       |
| 程序运行失败    | strace       | 定位缺失文件或库                 |
| 程序缺少库      | ldd          | 确认动态库依赖是否完整           |
| 没有硬件可用    | QEMU         | 模拟启动 rootfs，快速测试        |

------

### 11.6.8_小结

- **串口调试** 是基础，贯穿启动全流程；
- **strace/ldd** 是用户空间排查的利器；
- **QEMU** 适合无硬件情况下的验证；
- **日志系统** 提供额外的排错信息。

掌握这些工具后，可以大幅度缩短调试周期，并更系统地分析问题。

## 11.7_小结

------

### 11.7.1_本章回顾

在第十一部分中，我们聚焦于 **调试与问题排查**，把 Buildroot 从构建到运行过程中的常见问题，分阶段进行了系统化梳理：

- **11.1 引言**
  - 说明了问题可能出现的层次：构建、镜像生成、启动、用户空间。
- **11.2 构建阶段的常见问题与调试**
  - 下载失败 → 使用本地缓存、国内镜像；
  - 依赖缺失 → 安装宿主机依赖；
  - 编译错误 → 检查 GCC 版本、补丁应用；
  - 配置冲突 → 调整 `menuconfig`。
- **11.3 镜像生成与部署问题**
  - genimage 配置错误 → 检查 `genimage.cfg` 和分区大小；
  - 分区偏移错误 → 对照芯片手册确认 Bootloader 偏移；
  - 烧录失败 → 确认 `dd` 目标设备，验证镜像写入正确性。
- **11.4 启动阶段问题排查**
  - U-Boot 阶段：检查 Bootloader 写入位置、串口波特率；
  - Kernel 阶段：确认内核格式（zImage/uImage）、dtb、加载地址；
  - Rootfs 阶段：检查文件系统支持、bootargs、init 程序。
- **11.5 用户空间问题排查**
  - init 程序缺失或配置错误；
  - BusyBox 命令裁剪过多导致命令找不到；
  - 动态库依赖缺失（`ldd` 可检查）；
  - 配置文件/设备节点缺失。
- **11.6 调试工具与方法**
  - 串口日志：最基础的调试手段；
  - `strace`：跟踪系统调用；
  - `ldd`：确认动态库依赖；
  - `QEMU`：模拟启动；
  - `dmesg`/日志文件：辅助分析问题。

------

### 11.7.2_核心理念

1. **分层排查**
   - 构建 → 镜像生成 → 启动 → 用户空间
   - 每一层确认正确后再进入下一层。
2. **工具驱动**
   - 构建问题 → `make <pkg>-dirclean`、`make V=1`；
   - 启动问题 → 串口 log；
   - 用户空间问题 → `strace`/`ldd`。
3. **可重复性**
   - 保持构建环境一致，避免因为宿主机版本差异导致问题；
   - 使用 Buildroot 的 post-image 脚本生成校验和、签名，确保部署一致。
