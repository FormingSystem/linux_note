# 第1章_kref_引用计数机制章节大纲

本文档按当前 `kref-引用计数` 目录的实际写作进度重构。

前 3 章已经不只是“引入”，而是把很多核心边界都提前展开了：

```text
第 1 章：问题域和边界
第 2 章：源码入口和结构模型
第 3 章：生命周期状态机
```

所以后续章节不应该继续重复讲“kref 不等于锁”“get 前要证明对象有效”“put 后不能再访问”这些基础结论，而应该把它们当成前置原则，然后进入更具体的规则、API、lookup、锁、RCU 和工程模板。

一句话主线：

```text
kref 不是计数器 API，而是 Linux 内核中一种底层对象生命周期工具。

裸 kref 讲的是自定义对象如何管理引用；
device/class/bus 讲的是 driver core 已经封装好的分层对象模型。
```

## 1.2_kref_要解决什么问题

定位：建立问题域和边界。

本章已经承担这些职责：

```text
1. 裸指针共享为什么危险。
2. use-after-free 的本质。
3. kref 解决的是引用所有权，不是“有没有指针”。
4. kref 不解决并发互斥问题。
5. kref 不是设备安全代理，也不托管设备独占访问。
6. 生命周期保护和字段保护是两类问题。
7. kref 不能替代对象状态机。
8. kref 不能单独解决 lookup 问题。
9. handoff 是引用所有权转移问题。
10. 裸 kref 的 my_refobj 模型不能直接套到 device/class/bus 框架对象上。
```

本章后续如果继续修改，重点不是扩更多 API，而是把边界讲准：

```text
kref 保护对象内存生命周期；
锁/RCU/状态机/设备模型规则保护访问安全。
```

## 1.12_kref_refcount_t_kobject_的边界

定位：做概念对比，避免把几个层次混起来。

建议把原来的 “kref 不是 kobject” 扩成边界章节：

```text
1. atomic_t：底层原子计数工具。
2. refcount_t：引用计数安全原语。
3. kref：对象生命周期引用计数封装。
4. kobject：设备模型对象，不只是引用计数。
5. device/class/bus：driver core 的类型化对象和总分结构。
6. device_release/class_release 不是裸 kref 的 my_refobj_release。
7. 为什么不要为了引用计数强行引入 kobject。
```

这章只做对比，不展开 kobject 全体系；重点是避免把裸 kref 示例对象套到 driver core 框架对象上。

## 1.13_典型错误模式与调试线索

定位：错误清单，不再大段重复前文原理。

建议按 bug 类型组织：

```text
1. 少 get：异步路径 UAF。
2. 少 put：对象泄漏。
3. 多 put：提前 release / refcount underflow。
4. put 后访问：UAF。
5. handoff 后再 get：窗口已经错过。
6. lookup 后无保护 get：悬挂指针。
7. release 里重新发布对象：复活已销毁对象。
8. release 不脱链：全局结构留下悬挂指针。
9. 把 kref 当锁：字段竞争仍然存在。
10. 重复 kref_init：破坏已有引用关系。
```

本章可以加入：

```text
KASAN
KCSAN
lockdep
refcount warning
WARN_ON(!list_empty(...))
```
