# 第1章\_字符设备专题大纲

## 1.1\_专题定位

字符设备不是“申请设备号再填一张 `file_operations` 表”这么简单。它把用户空间文件描述符、VFS、字符设备映射、驱动对象和真实硬件连接起来。本专题维护字符设备自身的完整主线，同时在交叉点给出足以继续阅读的桥接解释。

VFS、设备模型、设备树、GPIO、IRQ、DMA 和等待通知分别建设独立权威专题，不能降格成字符设备的附属知识。VFS 的完整对象、挂载、路径、file、I/O 和回收体系见 [VFS 子系统专题](../../kernel_subsystems/vfs/大纲.md)；它们之间的职责和建设顺序见 [Linux I/O 与驱动子系统建设路线](../../../atlas/roadmaps/linux_io_driver_subsystems.md)。

```mermaid
flowchart LR
    U[用户系统调用] --> V[VFS 与 struct file]
    V --> M[dev_t 到 cdev 的映射]
    M --> F[file_operations]
    F --> D[设备对象与驱动状态]
    D --> H[硬件或软件数据源]
    H --> N[IRQ/DMA 更新完成状态]
    N --> W[等待队列/poll 通知进程]
    W --> U
```

## 1.2\_阅读阶段

### 1.2.1\_阶段一\_问题与对象模型

1. [字符设备最小模型](P01_字符设备最小模型.md)：从用户访问硬件的矛盾出发，分清设备号、设备节点、`cdev`、`inode`、`file` 与 `file_operations`。
2. [设备号、注册与设备节点](P02_设备号_注册与设备节点.md)：理解“内核可分派”和“用户可访问”是两件事。
3. [打开路径与文件操作](P03_打开路径与文件操作.md)：承接 VFS 的 [open 状态机](../../kernel_subsystems/vfs/P12_open状态机.md)，沿字符特殊文件分支建立 `inode->i_cdev`、`file->f_op` 与 `private_data` 的关系。

### 1.2.2\_阶段二\_贯通打开、I/O、通知与生命周期

4. [文件操作并发与生命周期](P04_文件操作并发与生命周期.md)：承接 VFS 的 [fd/file 生命周期](../../kernel_subsystems/vfs/P13_fd_table与file生命周期.md)，建立设备级、打开实例级、请求级和 VMA 级状态及并发边界。
5. [文件操作契约与数据路径](P05_文件操作契约与数据路径.md)：解释短传输、用户复制、文件位置、锁和 ioctl ABI。
6. [阻塞、poll 与异步通知](P06_阻塞_poll与异步通知.md)：贯通条件状态、等待者登记、IRQ/DMA 更新和唤醒方向。
7. [mmap 与跨文件生命周期](P07_mmap与跨文件生命周期.md)：在 VFS [文件 mmap 与 page fault](../../kernel_subsystems/vfs/P18_文件mmap与page_fault.md)基础上，解释设备映射为什么可能越过 fd 生命周期。
8. [设备模型与硬件子系统交接](P08_设备模型与硬件子系统交接.md)：连接 platform、设备树、GPIO、IRQ、DMA 与字符设备实例。
9. [安全移除、旧 fd 与模块卸载](P09_安全移除_旧fd与模块卸载.md)：完成停止发布、异步取消、唤醒和引用排空。
10. 通用依赖：[等待队列](../../linux/waiting_notification/P01_等待队列.md)、[生命周期集成专题](../../linux/object_lifetime/integration/大纲.md)。

### 1.2.3\_阶段三\_硬件交叉、落地与验证

11. [字符设备驱动模板](P10_字符设备驱动模板.md)：保留唯一权威实现模板，后续继续接入 platform、设备树、GPIO、IRQ 和 DMA。
12. [构建运行与验证](P11_构建运行与验证.md)：验证动态/静态设备号与自动/手工节点的组合。
13. [常见故障与排查](P12_常见故障与排查.md)：根据节点类型、设备号、模块引用和日志定位问题。

## 1.3\_边界与延伸

- i.MX6ULL 的 GPIO、设备树和 LED 组合属于[平台实验](../../../labs/platforms/nxp/imx6ull/drivers/P01_LED点灯/P03_字符设备_LED与设备树完整实验.md)。
- `poll` 的等待条件先学习[等待队列](../../linux/waiting_notification/P01_等待队列.md)，再回到字符设备并发章节理解接口交叉。
- VFS 的路径、open、fd、通用 read/write 和文件 mmap 语义只在[VFS 专题](../../kernel_subsystems/vfs/大纲.md)维护；本专题保留字符特殊文件的分派、设备状态和硬件生命周期交叉。
- `mmap` 会让内存对象越过 fd 的生命期，应结合[VFS 文件 mmap](../../kernel_subsystems/vfs/P18_文件mmap与page_fault.md)和对象生命周期专题阅读。
- DMA、IOMMU 和 IRQ 不是字符设备注册机制的组成部分，但它们可能是完整设备 I/O 链的一部分。本专题必须讲清交叉对象、状态所有权和通知链，再链接各自的权威专题。历史混合笔记已迁至[调查材料](../../../research/investigations/character_device/字符设备扩展接口历史草稿.md)，只用于追溯，不作为推荐实现。

## 1.4\_学习完成标准

读完后应能独立回答：

1. `register_chrdev_region()`、`cdev_add()` 和 `device_create()` 各自建立了哪一层关系？
2. 第一次打开设备节点时，VFS 怎样找到驱动的 `open()`？
3. `inode->i_cdev` 与 `file->private_data` 分别适合定位什么对象？
4. 多个 fd、`dup()`、`fork()` 和 `mmap()` 会共享哪些状态？
5. 初始化失败、设备移除和模块卸载时，资源应按什么顺序回滚？
