# 📘《书籍章节类输出标准（增强版）》

### 补充于：2025-10-09

### 适用范围：Linux 驱动 / 内核 / 设备树 / U-Boot / Buildroot 类书籍章节

------

## 一、语法讲解部分新增要求

1. **必须同时说明语法的作用与意义**
   - 每个语法关键字或属性都要解释它在系统中的真实用途，而非仅格式。
   - 必须指出它在系统中的实际作用，如“控制了哪类引脚”“决定哪个阶段的行为”“影响了哪部分驱动逻辑”。
   - 禁止只展示写法不解释用途。
2. **必须给出使用场景**
   - 每条语法都需配合具体示例说明使用场景，如“在 UART 初始化中如何用”“在休眠状态如何切换”等。
   - 若语法与低功耗、引脚复用、外设状态切换等相关，必须说明它在**驱动层的触发时机**（如 probe 阶段、suspend 阶段）。
3. **必须指出“不写会怎样”**
   - 每条属性要解释如果省略该字段，系统行为会如何（例如：使用默认配置、导致外设无法初始化、复用关系失效等）。
   - 若该语法为可选项，也应说明其默认行为。
4. **必须说明驱动中的使用逻辑**
   - 必须结合驱动层 API 或数据结构说明语法如何被使用，例如：
     - `pinctrl_select_state()` 切换状态；
     - `of_xlate()` 解析参数；
     - `pinctrl_lookup_state()` 查找状态。
   - 让读者能看到“语法在 DTS 中出现 → 被驱动解析 → 最终作用于寄存器”的全过程。

------

## 二、数据结构讲解部分新增要求

1. **语法与结构体一一对应**

   - 对每个 DTS 属性，列出其在内核中的落点结构体（如 `struct pinctrl_state`、`struct pinctrl_map`、`struct pinctrl_dev` 等）。
   - 给出相应的结构体定义及字段注释，说明它与 DTS 属性的绑定关系。

2. **讲清从 DTS → 内核对象 的映射路径**

   - 使用“结构映射图”或“表格”展示从 DTS 节点到内核数据结构的层级关系：

     ```
     device_node → platform_device → pinctrl → pinctrl_state → pinctrl_map → pinctrl_dev → 寄存器配置
     ```

   - 必要时用 Mermaid 流程图或 sequenceDiagram 表示解析顺序。

3. **数据结构仅为辅助，不喧宾夺主**

   - 只解释与 DTS 解析直接相关的成员，禁止过度展开驱动实现细节。
   - 所有结构体出现都要有语法对应点，做到一一匹配。

------

## 三、整体结构保持不变

仍遵循《📖 写书章节生成标准》的七段式结构：

> **主题引入 → 数据结构视角 → 开发者视角（语法与语义） → 用户视角 → 可视化图示 → 示例代码 → 调试与验证 → 小结**

但“语法讲解”部分必须包含以下四个层级：

> **语法说明 → 实际作用 → 使用场景 → 驱动对应逻辑**

------

## 四、编写风格与输出要求

- **风格：** 以“出版书籍”口吻书写，逻辑流畅、语气正式，不使用教学口语。
- **表达要求：**
  - 对关键语句使用 Markdown 表格或代码块展示；
  - 所有引用路径或函数使用反引号（` `）；
  - 所有 Mermaid 图必须符合 Typora 通用语法，并转义特殊字符；
  - 每节末尾要有“小结”概述关键点。
- **输出格式：**
  - 一级标题：章
  - 二级标题：节
  - 三级标题：子节
  - 代码与表格要带注释，图要有说明文字。

------

✅ 从本标准开始，今后的章节（如设备树 pinctrl 系列）将：

- 在讲解语法时 **同步说明语义、使用场景、驱动对应逻辑**；
- 在讲解结构体时 **保证每个 DTS 关键字都能在代码结构中找到落点**；
- 以“可出版教材”标准构建章节。
