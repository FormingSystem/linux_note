# 📘 技术书籍章节类写作要求（融合八步法 + 出版结构）

## 一、章节总体结构

每一章必须采用**出版级章节结构**，且严格遵循以下固定框架：

```
第N章　主题名称

章节内容说明（简要说明本章目标与内容范围）

一、是什么？（历史渊源、发展进程、定位）
二、干什么？（解决的问题与设计动机）
三、怎么实现？（底层原理与处理逻辑）
四、怎么用？（使用方式与实践步骤）
五、通用接口或工具方法表（接口/函数/命令一览 + 详细讲解）
六、区别 / 避坑 / 限制 / 注意点（对比与边界分析）
七、完整示例与讲解（可运行代码 + 逻辑分解）
八、小结（核心要点与知识回顾）
```

------

## 二、内容风格要求

| 项目           | 要求                                                         |
| -------------- | ------------------------------------------------------------ |
| **语气**       | 严谨技术书风格，不使用比喻或感性语言                         |
| **逻辑顺序**   | 严格“先概念→再动机→再原理→再用法→再示例”                     |
| **讲解深度**   | 从宏观机制 → 数据结构 → 函数路径，层层递进                   |
| **说明性**     | 每个新术语首次出现必须解释其作用与所在层级（用户态 / 内核 / 驱动） |
| **示例风格**   | 源码片段前附逻辑图或时序图，示例后附详细注解                 |
| **Mermaid 图** | 必须可在 Typora 11.9+ 正确渲染，所有特殊符号使用 HTML 实体转义（`<`、`>`、`"` 等） |
| **表格**       | 用于总结、对比、核对接口，保证阅读快照式理解                 |
| **分批输出**   | 若内容过长，必须在结尾标注“（未完待续）”，下一次接续前文层级继续 |

------

## 三、章节内部风格示例

### 章节标题区

```
# 第14章　poll 机制：从用户等待到驱动唤醒的完整链路
```

### 开场说明区

```
章节内容说明：
本章从用户态、内核态到驱动层三条视角，系统解释 poll 机制的设计目的、底层实现、等待队列与唤醒机制，
并以事件触发流程为主线展开完整分析。
```

### 八步主体区（示意）

```
一、是什么？（历史渊源、定位）
——讲发展、来源、设计背景

二、干什么？（解决的问题）
——讲痛点、设计目标、机制作用

三、怎么实现？（底层原理）
——讲数据结构、等待队列、唤醒路径、时序

四、怎么用？（用法与调用顺序）
——讲典型场景、操作顺序、注意顺序性

五、通用接口或工具方法表
——列出所有函数/命令/接口，逐项解释参数和使用场景

六、区别 / 避坑 / 限制
——对比相似机制、列出常见错误、注意事项

七、完整示例与讲解
——给出简化代码 + 对应图 + 逻辑解说

八、小结
——表格化核心知识点，便于快速复习
```

------

## 四、适用范围

本标准适用于以下类型章节：

- Linux 内核与驱动开发机制（如 poll、waitqueue、workqueue、pinctrl 等）
- Linux 子系统架构（如 platform_driver、device tree、GPIO、clk、devm 等）
- 并发与同步原语（如 mutex、spinlock、seqlock、RCU 等）
- 用户态与内核接口机制（如 select、epoll、ioctl、procfs、sysfs 等）
- 任何机制、概念、架构性主题（如 udev、udev事件流、设备模型、总线匹配）

------

## 五、图文融合规则

1. 每章至少包含 **1 张 Mermaid 流程图**（或时序图）；
2. 若涉及驱动调用链，推荐使用 `sequenceDiagram`；
3. 涉及层级结构，使用 `flowchart TD`；
4. 图中所有符号转义：
   - `"` → `"`
   - `<` → `<`
   - `>` → `>`
5. 图下方必须配简短说明文字（作用、结论）。

------

## 六、附加要求（针对开发者类内容）

| 要素             | 说明                                                         |
| ---------------- | ------------------------------------------------------------ |
| **数据结构视角** | 必须解释关键结构体成员及其语义（如 `struct file_operations.poll`、`struct wait_queue_head_t`） |
| **驱动视角**     | 必须讲解 `.poll`、`.read`、`.write` 与 `wake_up` 的协同      |
| **用户视角**     | 必须讲解 `poll()` 调用行为与 revents 检查逻辑                |
| **代码解说**     | 每段示例代码必须配“执行逻辑解释”或“关键点标注”               |
| **核对表**       | 每章末尾应附“交付核对表”用于检查实现完整性                   |

------

## 七、输出约束

- 输出时标题层级、编号必须严格对齐；
- 文字尽量用 **中文专业术语 + 英文关键字括号**；
- 章节之间的衔接要自然（前章概念→本章机制→下章扩展）；
- 若本章内容超长，必须在末尾加“（未完待续）”标记；
- 后续续写时直接续接上次小节编号。
