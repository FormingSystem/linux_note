# 文档写作输出规范类（面向“并发与竞争”主题）

## 0. 总目标

- 以**开发者视角为主**：告诉读者“这个模块是什么、解决什么、什么时候用、怎么用、能跟谁混搭、哪些坑”。
- **模块为纲，不以问题驱动**（避免跳来跳去）。
- **先概念后机制再装配**：前期不引入具体环境/SoC；中后期再接入项目与实战。
- 配套**表格体系**：每章固定的概念区分表 / 用法速览表 / 核对表；混搭篇另有混搭矩阵与禁配表。
- 语言风格：**中文教科书式，但不生硬**；术语先做“缓冲式介绍”，再逐步使用专业术语。

------

## 1. 全书结构（宏观节奏）

1. **序章/术语缓冲**（白话→专业术语映射，最小规则）
2. **篇一：可见性与顺序**（CPU↔CPU、CPU↔设备）
3. **篇二：互斥与临界区**（自旋、互斥、读写信号量）
4. **篇三：读多写少**（seqcount/seqlock、RCU）
5. **篇四：事件与等待**（waitqueue、completion）
6. **篇五：执行路径**（中断/软中断/线程化/工作队列、timer/hrtimer）
7. **篇六：设备交互**（文件操作并发、DMA 与一致性）
8. **篇七：生命周期与收尾**（devres/kref/get-put、有序停机剧本）
9. **篇八：混搭与装配**（混搭矩阵 + 与环境无关的可编译装配样例）
10. **附录**（术语索引、屏障速查、原语一览、停机清单）

> 环境/SoC/性能调优等**放到后段**或另册，不在初期章节出现。

------

## 2. 章节内固定结构（每章统一骨架）

1. 〔概念〕：先用白话解释，再给**术语最小定义**。
2. 〔能做/不能做〕：明确能力与边界。
3. 〔核心用法模式〕：2–3 个“可直接套用”的**模式**（伪代码或最小片段）。
4. 〔混搭与边界〕：能与哪些模块组合、顺序与锁序规则、边界条件。
5. 〔常见坑〕：列举 4–6 条“看到就该改”的反模式并给对照修复。
6. 〔最小模板〕：**可粘贴**的小段（与环境无关）。
7. 〔交付核对表〕：上线前逐项勾选。

> **测试/验证视角**：只在必要点“点到为止”（例如是否需要 `_sync` 取消、是否需要 `wmb()`），不喧宾夺主。

------

## 3. 表格体系（强制插入点）

- **每章三张表**（固定位置与名称）
  1. 〔概念区分表〕（开头，1 页内）：相似/差异/边界/典型误解。
  2. 〔用法速览表〕（核心用法前）：能做/不能做、上下文限制、是否可睡、是否自带顺序、推荐组合。
  3. 〔核对表〕（章末，可打印）：关键动作与常见遗漏。
- **每篇一张总览表**：该篇所有模块横向对比（能力/限制/放置点/与相邻模块关系）。
- **混搭篇两张表**：
  - 〔混搭矩阵〕（宜配/慎配/禁配，附原因）
  - 〔禁配对照表〕（错误组合→后果→替代方案）

------

## 4. 术语与注释约定

- 术语首次出现**先白话后术语**，之后统一使用专业术语。
- 代码/片段注释标记（便于审计与复用）：
  - `[INV]` 不变量（核心约束）
  - `[MIX]` 混搭提示（与其它模块协同的注意点）
  - `[PIT]` 避坑（反模式或危险调用）
  - `[CHECK]` 核对项（交付要勾的点）

------

## 5. 取舍与节制

- **不做**：API 大抄本、问题驱动的碎片化叙述、过早引入环境细节。
- **要做**：少量、可直接上手的**模式/模板**；把 **边界与混搭** 讲清。

------

## 6. 章节示例（模板占位名，非环境绑定）

- 例：I/O 顺序章的最小模板

  ```c
  /* [INV] 设备必须先见到配置，再见到启动 */
  writel_relaxed(cfg0, base + REG_CFG0);
  writel_relaxed(cfg1, base + REG_CFG1);
  wmb(); /* [CHECK] 确认点：CPU→设备顺序保证 */
  writel(CTRL_EN, base + REG_CTRL);
  ```

- 例：等待队列章的最小模板

  ```c
  /* 条件函数在保护内评估 */
  if (wait_event_interruptible(q, cond_in_guard()))
      return -ERESTARTSYS;
  /* 醒后在保护内重检，再搬运到保护外 */
  ```

------

## 7. 最终交付形态（每章产出 checklist）

- PDF/Markdown 章节文档
- 三张固定表格（概念区分、用法速览、核对）
- 最小模板代码块（可拷贝）
- 关键混搭提示与禁配项（若适用）

------

## 8. 可粘贴的「写作提示词」（以后开写时直接贴给我）

**粘贴区开始**
 你要按“模块中心、开发者视角”的**输出规范类**写文档：

- 先概念后机制再装配，不引入具体环境/SoC，直到后段。
- 每章严格包含：〔概念〕→〔能做/不能做〕→〔核心用法模式〕→〔混搭与边界〕→〔常见坑〕→〔最小模板〕→〔交付核对表〕。
- 每章**必须**插入三张表：〔概念区分表〕、〔用法速览表〕、〔核对表〕；篇首提供总览表；混搭篇有混搭矩阵与禁配表。
- 术语第一次出现先白话解释，再使用专业术语；代码注释用 `[INV]/[MIX]/[PIT]/[CHECK]` 标记。
- 避免问题驱动的碎片化叙述；以模块为纲，说明该模块**怎么用、注意什么、能与谁混搭**。
   **粘贴区结束**

```yaml
doc_style:
  viewpoint: "developer-first"
  organization: "module-centric"
  progression: ["concepts", "patterns", "mechanisms", "assembly", "environment-late"]
  environment_intro: "late"
  terminology:
    first_use: "plain-explain-then-term"
    code_tags: ["[INV]", "[MIX]", "[PIT]", "[CHECK]"]
tables:
  per_chapter:
    - "概念区分表"
    - "用法速览表"
    - "核对表"
  per_part: "篇级总览表"
  mix_chapter:
    - "混搭矩阵"
    - "禁配对照表"
chapter_template:
  sections:
    - "概念"
    - "能做/不能做"
    - "核心用法模式"
    - "混搭与边界"
    - "常见坑"
    - "最小模板"
    - "交付核对表"
constraints:
  avoid: ["API大抄本", "问题驱动叙述", "过早引入环境"]
  must: ["模式/模板可直接套用", "明确边界与混搭"]
deliverables:
  - "Markdown或PDF章节"
  - "三张固定表格"
  - "可拷贝的最小模板代码"
```
