# 📖 写书章节生成标准（适用于 Linux 驱动/内核书籍类内容）

## 1. 输出方式

- 每次输出必须是一个 **完整章节**，章节号、标题、子标题要 **严格对齐**（例如“第 1 章 → 1.1 → 1.1.1 …”）。
- 如果算力不足，允许 **分批次输出**：
  - 每批次必须保证完整的小节，不要截断。
  - 明确标注“（未完待续）”，下一次从对应小节继续。
- 这样保证 **章节完整性** 和 **阅读连贯性**。

------

## 2. 内容结构

- 每章必须包含以下核心模块：

1. **主题引入**
   - 本章要解决的核心问题是什么？
   - 为什么对驱动开发者和用户重要？
2. **数据结构视角**
   - 详细解释相关 **内核数据结构**（字段、作用）。
   - 如果需要，分成 **原理视角** 和 **源码调用链视角**。
3. **开发者视角**
   - 说明开发者如何在驱动里调用 API（例如 class_create、device_create）。
   - 提供完整的 **最小化驱动示例**，能在现代内核中编译运行。
   - 示例必须覆盖本章的核心主题（如 sysfs 属性、class、uevent）。
4. **用户视角**
   - 展示用户空间如何操作（命令行示例：`ls`、`cat`、`echo`）。
   - 展示如何通过 `dmesg` 验证驱动是否响应。
   - 解释 `/sys` 和 `/dev` 的关系（如果涉及）。
5. **可视化图示**
   - 至少一种图示：
     - **目录树**（展示 sysfs 结构）
     - **流程图 flowchart**（展示调用链）
     - **时序图 sequenceDiagram**（展示事件交互）
   - 图必须与文字挂钩，帮助理解。
6. **调试与验证**
   - 提供可操作的验证方法。
   - 提示常见错误与排查思路。
7. **小结**
   - 概要本章内容，形成对比表格或要点。
   - 必须有“一句话总结”，强化记忆。

------

## 3. 主题化写作

- 每章只聚焦 **一个主题**，不要混杂多个主题。
- 主题要在开头 **明确提出**，中间通过 **数据结构、开发者实践、用户操作** 来逐层展开，最后总结。

------

## 4. 多视角要求

- **用户视角**：他们能看到什么？在 `/sys`、`/dev` 怎么验证？
- **开发者视角**：要写哪些代码？API 怎么用？回调怎么写？
- **内核视角（可选）**：源码链路或配置项的背景介绍。

------

## 5. 额外要求

- **分批次输出**：如果内容过长，允许拆分为多批，保证每次输出完整小节。
- **解锁算力限制**：不允许偷工减料或只给片段，必须保证书籍风格。
- **表格/代码/图示**：每章至少包含其中两类，避免纯文字枯燥。
- **mermaid输出要求**：mermaid输出的渲染软件为typora，采用通用语法，不要使用方言，保证语法正确。

------

## ✅ 示例指令（以后可直接用）

> 按照“写书章节生成标准”，给我输出 **第 X 章：XXXX**。
>
> - 保证章节完整，标题层级严格对齐。
> - 必须包含：主题引入 + 数据结构解析 + 开发者视角 + 用户视角 + 图示 + 示例代码 + 调试验证 + 小结。
> - 如果内容过长，请分批次输出，保证每次是完整小节。
> - 需要从 **用户/开发者/内核多视角** 展开，不要只写单一角度。
