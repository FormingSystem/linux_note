## 输出章节类统一要求（最新版）

1. **分批输出原则**
   - 如果内容一次写不完，必须分批次输出。
   - 每一批都要是“完整的小节”，不能只给提纲。
   - 批次末尾标注“（未完待续）”或说明后面还有 12.x。
2. **结构固定（书稿级）**
    每一章/大节按这个顺序写：
   1. 章节引入 / 背景 / 本章目标
   2. 数据结构视角（内核里是哪几个 struct / 域 / 描述符）
   3. 开发者视角（驱动怎么写、probe 怎么下、exit 怎么退、devm 版本要写）
   4. 用户/平台视角（DTS/ACPI 怎么配、不写的后果、常见错误）
   5. 可视化图示（Mermaid 必须可在 Typora 渲染）
   6. 示例代码（能直接粘到驱动/内核里跑的模板，C 为主）
   7. 调试与验证（dmesg、/proc/interrupts、sysfs、常见坑）
   8. 小结（和前后章节的勾连：比如和层级中断/PM/多核的关系）
3. **Mermaid 特殊要求**
   - 保证能在 Typora (11.9+) 渲染。
   - 若语句复杂，用 `"` 包外层，内部统一用 `"`。
4. **语法讲解必须“四连问”**
    每遇到一个 DTS 属性 / 一个内核接口 / 一个语法，必须说明：
   - 它的**作用**是什么；
   - **使用场景**是什么；
   - **不写/写错的后果**是什么；
   - **在驱动里的使用逻辑/落点**是什么。
5. **devres 必须带上**
   - 讲中断、GPIO、clk、pinctrl、MSI、irq_domain 时，都要同时说非 devm_ 和 devm_ 的版本；
   - 要说明 devm_ 的释放逻辑、适用场景、和老接口的区别；
   - 不是让一律用 devm_，而是必须“讲到”。
6. **不要用比喻**
   - 解释要用严格技术表述：定义、流程、边界、示例；
   - 不用生活类比、不用形象化说法。
7. **和前面章节保持口径**
   - 中断相关要能和你前面第3章（irq_domain）、第5章（GPIO/触发）、第9章（层级域）、第11章（PM/唤醒）对上；
   - 如果是中断类内容，默认平台是你说的 i.MX6ULL / GICv2/v3 / 6.1+ 内核；
   - 举 DTS 例子时优先用你之前说的 `demo_led_key_int@0` 那个节点做参考。
8. **可直接入书**
   - 用出版风格中文标题：`第12章 …` → `12.1 …` → `12.1.1 …`
   - 段落完整、表格清晰、代码高亮、可以直接贴进 Typora/WPS。
