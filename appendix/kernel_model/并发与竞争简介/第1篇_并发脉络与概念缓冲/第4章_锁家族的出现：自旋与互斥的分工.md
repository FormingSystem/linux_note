# 第4章　锁家族的出现：自旋与互斥的分工

------

## 章节内容说明

前一章讲到“等待与唤醒”的事件机制，为线程间协作提供了语义层支持；
 但在多核并发环境中，仅仅依靠等待/唤醒无法防止**共享数据被同时修改**。

本章介绍“锁（Lock）”机制的出现及其分化：

- 为什么需要自旋锁（spinlock）；
- 为什么又出现了互斥锁（mutex）与读写信号量（rwsem）；
- 这些锁各自解决了什么问题、在哪些场合不能混用。

本章不讲 API，仅讲**机制逻辑与上下文限制**。

------

## 4.1　自旋锁：短临界区的原子保护

### 概念

**自旋锁（spinlock）\**是一种在不可睡眠上下文中使用的锁。
 当一个 CPU 想获取被占用的锁时，它会\**反复检测锁状态**，直到持锁方释放。

```c
while (lock == BUSY)
    cpu_relax();    /* [PIT] 忙等等待锁释放 */
```

### 解决了什么问题

- 为中断上下文、软中断、底半部等**不可睡眠场景**提供了安全的互斥机制；
- 消除了“睡眠等待”带来的调度切换开销；
- 适合保护**执行时间极短的临界区**。

### 带来了什么新问题

| 问题类型       | 描述                                       |
| -------------- | ------------------------------------------ |
| 忙等消耗       | 在持锁时间稍长的情况下浪费 CPU             |
| 中断死锁       | 若锁在中断中被占用，持锁路径再被中断将自锁 |
| 缺乏优先级继承 | 高优先级任务可能被低优先级任务长期阻塞     |
| 无法睡眠       | 禁止使用任何可能导致调度的操作             |

------

### 表 4-1　自旋锁使用特征

| 特征         | 值                                 |
| ------------ | ---------------------------------- |
| 是否可睡     | 否                                 |
| 上下文限制   | 可在中断 / 原子上下文使用          |
| 开销类型     | 忙等时间                           |
| 推荐使用场景 | 中断服务、底半部、快速共享状态更新 |
| 禁用场景     | 用户态路径、长耗时操作             |

------

## 4.2　互斥锁与读写信号量：长操作的协调

### 概念

**互斥锁（mutex）**允许任务在等待锁时睡眠，让出 CPU；
 **读写信号量（rwsem）**则进一步区分读共享与写独占。

```c
mutex_lock(&m);     /* [INV] 可睡眠锁 */
critical_section();
mutex_unlock(&m);
```

### 解决了什么问题

- 支持**长耗时的临界区**操作；
- 自动调度等待者，不浪费 CPU；
- 支持优先级继承，避免“优先级反转”。

### 带来了什么新问题

| 问题类型         | 描述                         |
| ---------------- | ---------------------------- |
| 不可在中断中使用 | 因为可能睡眠                 |
| 上下文切换成本   | 调度切换与唤醒开销较大       |
| 死锁风险         | 多锁嵌套或顺序错误易形成死锁 |
| 信号唤醒竞态     | 等待队列操作需与条件判断同步 |

------

### 表 4-2　自旋锁 / 互斥锁 / 读写信号量对照表

| 类型       | 是否可睡 | 典型场景           | 优点         | 限制                 |
| ---------- | -------- | ------------------ | ------------ | -------------------- |
| 自旋锁     | 否       | 中断、底半部       | 低延迟、轻量 | 忙等、短临界区限定   |
| 互斥锁     | 是       | 驱动控制路径       | 可睡、易用   | 不能用于中断上下文   |
| 读写信号量 | 是       | 大数据结构读多写少 | 支持并发读   | 写路径复杂、唤醒竞态 |

------

## 4.3　锁序与嵌套：从局部互斥到系统死锁

### 概念

锁并发使用时，若多个锁的获取顺序不一致，就可能产生**死锁（deadlock）**：

> 线程 A 持有锁 L1 等待 L2，而线程 B 持有 L2 等待 L1。

------

### 防范策略

1. 明确**锁获取顺序**；
2. 避免**多锁交叉持有**；
3. 对资源使用**层级编号（lockdep）**；
4. 对嵌套持锁函数使用 `*_nested()` 系列接口。

------

### 表 4-3　典型死锁情境与修复

| 场景          | 触发条件               | 修复方案                         |
| ------------- | ---------------------- | -------------------------------- |
| 双锁交叉      | 两线程锁顺序相反       | 统一锁顺序                       |
| 自旋+互斥交叉 | 自旋内调用可能睡眠函数 | 拆分临界区                       |
| 中断重入      | ISR 持锁后再次触发中断 | 屏蔽中断或使用 spin_lock_irqsave |

------

## 4.4　混搭与边界

| 组合                | 是否安全       | 原因                               |
| ------------------- | -------------- | ---------------------------------- |
| 自旋锁 + 中断       | 安全，常见组合 | 必须使用 irqsave/restore 版本      |
| 自旋锁 + 睡眠操作   | 禁止           | 违反上下文约束                     |
| 互斥锁 + 等待队列   | 安全           | 两者均可睡眠                       |
| 自旋锁 + RCU        | 谨慎           | RCU 读端可与自旋重叠，但写端需同步 |
| 信号量 + completion | 一般不推荐     | 二者语义重叠，易混淆唤醒时机       |

------

## 4.5　常见坑

| 标识   | 描述                                             |
| ------ | ------------------------------------------------ |
| [PIT1] | 在中断中调用 `mutex_lock()`                      |
| [PIT2] | 自旋锁保护过大临界区                             |
| [PIT3] | 未关闭中断即持锁                                 |
| [PIT4] | 多锁嵌套无序                                     |
| [PIT5] | 互斥锁在错误路径未解锁                           |
| [PIT6] | 忘记使用 `_irqsave/_irqrestore` 导致中断嵌套死锁 |

------

## 4.6　最小模板

```c
/* [INV] 自旋锁保护共享标志 */
spin_lock_irqsave(&lock, flags);
shared_flag = 1;
spin_unlock_irqrestore(&lock, flags);

/* [INV] 可睡眠互斥锁保护复杂操作 */
mutex_lock(&m);
update_device_config();
mutex_unlock(&m);

/* [CHECK] 验证：短临界区采用自旋，长临界区采用互斥 */
```

------

### 表 4-4　核对表

| 核对项 [CHECK]               | 说明                 |
| ---------------------------- | -------------------- |
| 是否在中断中使用了可睡眠锁？ | 禁止                 |
| 自旋锁是否用于短临界区？     | 否则浪费 CPU         |
| 是否按固定顺序获取多把锁？   | 否则风险死锁         |
| 是否使用 irqsave/restore？   | 保护中断上下文一致性 |
| 是否区分控制面与数据面？     | 便于锁粒度优化       |

------

## 4.7　小结

1. **自旋锁**用于短路径、不可睡上下文；
2. **互斥锁**与**读写信号量**用于长路径、可睡上下文；
3. 锁序与上下文是驱动安全性的核心约束；
4. Linux 锁家族的出现，标志着从“临界区互斥”到“系统化同步”阶段的转变。
