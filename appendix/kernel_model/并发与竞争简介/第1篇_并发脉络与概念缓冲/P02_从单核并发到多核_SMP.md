# 第2章　从单核并发到多核 SMP

------

## 章节内容说明

本章承接前文的中断机制，进入并发问题的第二个转折点——**多核（SMP, Symmetric Multi-Processing）架构的出现**。
 在单核时代，所有代码共享同一执行流；而在多核时代，多个 CPU 核心可并行访问同一内存区域，这使得“代码顺序执行”与“硬件执行顺序”开始出现分离。

我们将明确以下三个关键转变：

1. 从时间片并发到**真正的物理并发**；
2. 从逻辑顺序到**内存可见性模型（Memory Model）**；
3. 驱动中必须显式定义“谁先、谁后”的顺序关系。

------

## 2.1　多核登场：并行吞吐与三类新错位

### 概念

**SMP 架构**允许多个 CPU 核心共享同一物理内存，每个核心拥有自己的缓存层（L1/L2）。
 这意味着：多个执行流可能同时访问同一地址，但它们各自持有的缓存副本未必同步。

### 解决了什么问题

- **性能扩展**：多个核心可同时处理不同任务或同一任务的不同部分；
- **中断分布**：可将中断负载分配给不同 CPU，减少延迟；
- **资源复用**：共享内存模型简化了系统架构。

### 带来了什么新问题

| 问题类型       | 描述                                       |
| -------------- | ------------------------------------------ |
| **互斥问题**   | 多个核心同时写入同一数据，产生写写冲突     |
| **顺序问题**   | CPU 内部乱序执行导致“看见顺序 ≠ 程序顺序”  |
| **可见性问题** | 核心 A 写入缓存未同步到主存，核心 B 看不到 |

------

### 表 2-1　SMP 带来的三类错位

| 错位类型   | 触发场景         | 典型现象           | 对应解决机制              |
| ---------- | ---------------- | ------------------ | ------------------------- |
| 互斥错位   | 多核写同一变量   | 数据覆盖、状态错乱 | 锁与原子操作              |
| 顺序错位   | CPU 执行顺序重排 | 判断条件提前成立   | 内存屏障（barrier）       |
| 可见性错位 | 缓存未同步       | 他核读到旧值       | 缓存一致性协议 / 显式同步 |

------

## 2.2　内存模型意识：显式顺序的诞生

### 概念

在单核系统中，程序员默认“代码的书写顺序 == 执行顺序”。
 然而在现代 CPU 中，为了优化性能，指令可被乱序执行；缓存、写缓冲区也会延迟可见性。
 因此，Linux 与硬件共同引入了**内存模型（Memory Model）\**与\**显式屏障（barrier）机制**。

------

### 解决了什么问题

- 为“先有 A 后有 B”提供形式化定义（happens-before 关系）；
- 允许开发者控制“何时刷新缓存、何时同步视图”；
- 建立了跨核心通信的**可见性契约**。

------

### 带来了什么新问题

| 问题       | 说明                   |
| ---------- | ---------------------- |
| 屏障过多   | 性能下降，增加延迟     |
| 屏障缺失   | 数据乱序，状态未更新   |
| 难以验证   | 顺序错误难复现、难调试 |
| 可读性下降 | 程序逻辑被技术细节掩盖 |

> **要点**：内存模型并不自动保证正确性，它只是提供“控制顺序的工具”。

------

### 表 2-2　CPU ↔ CPU 间的顺序与可见性特征

| 类别           | 保证对象      | 是否自动有序   | 是否需显式屏障 | 常见场景     |
| -------------- | ------------- | -------------- | -------------- | ------------ |
| 普通内存访问   | CPU 缓存/主存 | 否             | 是             | 驱动数据共享 |
| I/O 寄存器访问 | MMIO 区域     | 否             | 是             | 外设配置顺序 |
| 同步原语       | 锁、原子操作  | 是（隐式屏障） | 否             | 互斥、同步   |

------

## 2.3　驱动视角下的影响

### 稳定性

- 不加控制的多核访问可能导致**竞态条件**（Race Condition）；
- 驱动中任何共享数据结构必须有明确的**访问策略**（锁/屏障）。

### 尾延迟

- 过多锁与屏障增加延迟，降低中断响应速度；
- SMP 调度切换可能导致**cache bounce**，即频繁缓存迁移。

### 可维护性

- “看似顺序”但“实则乱序”的代码会让问题在现场极难复现；
- 推荐在关键通信点添加**显式内存屏障注释**。

------

### 表 2-3　核对表

| 核对项 [CHECK]                 | 说明                       |
| ------------------------------ | -------------------------- |
| 是否有跨 CPU 共享的变量？      | 必须受保护或加屏障         |
| 是否假设书写顺序即执行顺序？   | 需显式 barrier 保证        |
| 是否对关键路径过度加锁？       | 评估延迟与开销             |
| 是否在 SMP 测试环境验证？      | 单核验证不足以发现乱序问题 |
| 是否标注了通信点的可见性要求？ | 推荐文档化 barrier 语义    |

------

## 2.4　小结

1. **多核系统打破了“顺序即因果”的假设**；
2. 程序顺序、执行顺序、观察顺序三者分离，形成了并发错误的温床；
3. 从此之后，“锁、屏障、原子操作”成为驱动并发编程的基础工具。
