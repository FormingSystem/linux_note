# 第1章：GPL 协议说明（总论）

> 目标：用工程视角说明 GPL 家族的范围、核心义务、触发条件、与常见工程场景（内核、库、SaaS、固件、嵌入式）的关系，并给出合规落地方法与检查清单。本章不使用比喻。

------

## 1.1 名称与范围

- **GPL 家族**：GPLv1（历史）、**GPLv2**、**GPLv3**；相关弱版权与网络传播扩展：**LGPLv2.1 / LGPLv3**、**AGPLv3**。
- **版权属性**：**Copyleft**（传播性版权）。在满足触发条件的分发下，派生作品需在相同或兼容条款下提供**对应的源代码**与相应权限。
- **文本来源**：自由软件基金会（FSF）发布的标准许可文本。工程中应引用**原文**并保留全文。

## 1.2 版本差异概览（工程要点）

| 许可            | 主要对象       | 触发范围                                     | 专利条款               | 设备锁定/安装信息                      | 典型用途                  |
| --------------- | -------------- | -------------------------------------------- | ---------------------- | -------------------------------------- | ------------------------- |
| **GPLv2**       | 程序/内核/应用 | 复制、分发、改作                             | 无明确授予条款         | 无规定                                 | Linux 内核、传统 GPL 项目 |
| **GPLv3**       | 程序/内核以外  | 复制、分发、改作；含 DRM/规避法条款          | 明确专利授予与反诉终止 | 要求提供“安装信息”（Anti-Tivoization） | 工具链、应用、固件等      |
| **LGPLv2.1/v3** | **库**         | 链接关系下的弱版权                           | 与对应 GPL 版本一致    | 与对应 GPL 版本一致                    | 动态库供闭源应用链接      |
| **AGPLv3**      | 网络服务       | **通过网络提供给用户使用**即触发（远程交互） | 同 v3                  | 同 v3                                  | SaaS/Server 侧程序        |

## 1.3 触发条件与义务（按场景分类）

### 1.3.1 一般分发（GPLv2/v3 共性）

当你**分发**可执行文件或其副本（包括固件镜像、安装包、设备内预装）：

- **必须**：提供或附带**对应源码**（Corresponding Source，含构建脚本、接口定义、补丁）。
- **源代码可用性**：随二进制一并提供，或以书面要约方式提供**至少 3 年**有效期（v2/v3）。
- **许可一致性**：派生作品继续在 GPL 下；不得增加限制原有权利的额外条款（除非“额外权限”例外）。
- **版权与许可声明**：保留原有版权与许可文本；在关于框、帮助、README 中标注。

### 1.3.2 链接与派生判定（库与应用）

- **GPL（强 copyleft）**：静态或动态链接一般被视为**同一程序的派生作品**；分发时整体验证 GPL 义务。
- **LGPL（弱 copyleft）**：允许闭源程序链接库；要求可替换库（例如提供目标文件以便重新链接，或采用动态链接方式）；对库本身的修改需在 LGPL 下开放源码。
- **AGPL**：即使不分发二进制，只要**通过网络向外提供服务**，也需向与之交互的用户提供对应源码（远程交互条款）。

### 1.3.3 设备与安装信息（仅 GPLv3/AGPLv3）

- 对于“用户产品”（可由终端用户使用的设备），如果分发了受 GPLv3 约束的二进制，**需提供安装信息**（包括用于安装、授权运行修改后版本的密钥/流程）。
- 防篡改技术（DRM）不得阻止用户在其设备上运行修改版本（法律合规模块另计）。

### 1.3.4 专利与反诉（GPLv3/AGPLv3）

- 明确**专利许可授予**：分发者对自身可实施的必要专利授予使用者非排他许可；
- 反诉终止：对项目或下游提起专利诉讼可触发许可终止。

------

## 1.4 Linux 内核特例（GPLv2-only）

- **内核许可**：Linux 内核为 **GPLv2-only**，非 “v2 or later”。
- **内核模块**：
  - **GPL-only 符号**：部分导出符号（`EXPORT_SYMBOL_GPL`）仅允许标注为 GPL 的模块使用；
  - **模块声明**：`MODULE_LICENSE("GPL")` 或兼容字符串；错误声明可能导致符号解析失败或警告；
  - **派生性判断**：静态链接/紧密耦合的内核模块通常被视为派生作品，需遵循 GPLv2。
- **用户空间接口**：通过内核系统调用/设备文件进行交互的用户态程序**不因调用内核接口而受 GPL 约束**（接口例外）。

------

## 1.5 与其他许可的兼容性

- **MIT/BSD/ISC → GPL**：可并入 GPL 项目（兼容向 GPL 聚合）；分发时整体按 GPL 提供。
- **Apache-2.0 → GPL**：与 **GPLv3** 兼容；与 **GPLv2-only** 不兼容（除非获得额外权限或双许可）。
- **LGPL → 商业闭源**：可链接（动态优先）；需确保替换能力与库修改的开源义务。
- **AGPL**：与传统闭源 SaaS 模型冲突较大；如采用需评估远程交互条款的源代码提供义务。

------

## 1.6 工程交付要求（最小合规集）

1. **在仓库与发行物中包含**：
   - `COPYING` 或 `LICENSE`（完整文本，对应版本）；
   - 每个源文件头的 SPDX 标识（示例：`// SPDX-License-Identifier: GPL-2.0-only`）；
   - `NOTICE`/`ACKNOWLEDGMENTS`（如有第三方致谢要求）。
2. **分发二进制时提供源码**：
   - 同介质附带源码，或提供可信的下载地址；
   - 如采用书面要约（v2/v3），保证**三年有效**；
   - 对 GPLv3“用户产品”，同时提供**安装信息**。
3. **构建复现**：
   - 包含 `Makefile`/`Kconfig`/构建脚本；
   - 记录工具链版本；
   - 对补丁集提供增量或合并后的源码。
4. **第三方许可证清单**：
   - 列出所用第三方依赖、版本、许可证、源码获取方式。
5. **模块/库边界**：
   - 内核模块使用 `EXPORT_SYMBOL_GPL` 的符号需确保模块为 GPL 兼容；
   - 对 LGPL 库的使用确保可替换性（动态链接或提供目标文件）。
6. **专利与合规流程**：
   - 评估是否存在专利承诺或交叉许可需求（GPLv3/AGPLv3 下尤需关注）；
   - 建立违规修复与下游通知流程（合规事件响应）。

## 1.7 常见工程问题（FAQ）

**Q1：只在内部使用，不分发，是否需要开源修改？**
 A：GPL 的主要义务在**分发时**触发。内部使用且未分发，不触发源代码提供义务（AGPL 对外提供网络服务除外）。

**Q2：通过 IPC/REST 与 GPL 程序通信，会被“感染”吗？**
 A：单纯进程间通信或网络协议通常不被视为同一程序的派生作品；但若存在密切耦合或规避意图，需要法律评估。

**Q3：动态链接 LGPL 库需要开放自身源码吗？**
 A：不需要。前提是满足**可替换**要求（提供重新链接手段或使用动态链接），且对库本身的修改需在 LGPL 下开放。

**Q4：基于 GPLv2 的内核，驱动模块能闭源吗？**
 A：存在合规与合规性风险。使用 `EXPORT_SYMBOL_GPL` 的符号要求 GPL 模块；即使未使用该类符号，模块是否构成派生作品仍有争议。工程上建议采用 **GPL 兼容许可** 或与权利人达成例外/商业授权。

**Q5：对 GPLv3 设备是否必须提供签名密钥？**
 A：需要提供“安装信息”以使用户能安装并运行修改版本。若通过签名机制限制运行，则需提供用户可用的签名流程/密钥或等效手段。

## 1.8 项目落地模板

- **SPDX 头**（每个源文件）
   `/* SPDX-License-Identifier: GPL-2.0-only */`
- **仓库根目录**
  - `LICENSE`：对应 GPL 文本；
  - `THIRD_PARTY_LICENSES/`：依赖清单与许可证副本；
  - `SOURCE_OFFER.txt`（如提供三年书面要约）；
  - `BUILDING.md`：构建环境、脚本、工具链版本；
  - `COMPLIANCE.md`：合规流程、联系方式、安装信息（v3）。
- **发行物**
  - 二进制包与对应源码包/链接；
  - 安装信息（v3/AGPLv3 适用）；
  - 变更记录与补丁说明。

## 1.9 变更与升级策略

- **版本锁定**：确认上游是 `-only` 还是 `-or-later`（例如 Linux 内核是 `GPL-2.0-only`）。
- **双许可**：在自有代码上可采用双许可（例如 GPLv2 + 商业许可），以适配不同分发模式。
- **依赖替换**：若 AGPL 触发网络源代码义务且与业务不兼容，考虑替换为 LGPL/MIT 组件。
- **合规自动化**：引入 SPDX 扫描、依赖分析（SBOM）、许可证校验（CI Gate）。

------

## 1.10 合规检查清单（交付前执行）

-  明确项目适用许可（版本精确：`-only` / `-or-later`）。
-  仓库包含完整许可文本、SPDX 标识、第三方清单。
-  若分发二进制，确保**对应源码**与**三年要约**（如适用）。
-  GPLv3/AGPLv3：提供安装信息/网络源代码访问路径。
-  内核/模块：核对 `EXPORT_SYMBOL_GPL` 使用与 `MODULE_LICENSE`。
-  LGPL 链接场景：验证动态链接或可重新链接能力。
-  专利与反诉条款评估（GPLv3/AGPLv3）。
-  交付介质与 README 中明确用户获取源代码的方法与联系方式。
-  预置合规响应流程与修复 SLA。

### 本章小结

- GPL 家族以**分发触发义务**为核心，要求在触发时提供**对应源码**与**相同许可**。
- GPLv2 与 GPLv3 在**专利**与**设备安装信息**方面差异显著；AGPL 将触发范围扩展到**网络提供**。
- Linux 内核为 **GPLv2-only**，模块与 GPL-only 符号需严格对齐。
- 工程合规的关键是：**版本精确、触发识别、源码可得、安装信息、第三方清单、CI 校验**。
