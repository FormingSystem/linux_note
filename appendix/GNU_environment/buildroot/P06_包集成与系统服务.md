## 为什么需要包集成与系统服务

在前面的章节中，我们已经通过 Buildroot 构建了内核和根文件系统，并学会了如何裁剪和定制 rootfs。
 然而，一个实际可交付的嵌入式产品，远远不止一个精简的 BusyBox：

- 它可能需要 **图形界面库**（如 Qt、GTK）。
- 它可能需要 **网络协议栈扩展**（如 OpenSSL、MQTT 库）。
- 它可能需要 **系统服务**（如网络守护进程、日志服务、监控守护进程）。
- 它可能需要 **自研应用**（如设备控制逻辑、UI 程序）。

这些额外的库和服务，必须与 Buildroot 集成，才能在最终固件中体现出来。

------

## Buildroot 的“包”概念

在 Buildroot 中，**包 (package)** 是最小的构建单元。

- 每一个外部库、工具或应用，都会对应一个包。
- 包定义了它的 **源码来源、依赖关系、构建方式、安装路径**。
- 通过选择合适的包组合，开发者可以拼装出一个满足产品需求的 rootfs。

例如：

- `busybox` → 提供基础命令。
- `openssl` → 提供 TLS 支持。
- `dropbear` → 提供轻量级 SSH 服务。
- `qt5` → 提供图形界面支持。

------

## 系统服务的重要性

在嵌入式系统中，rootfs 启动后必须进入一个稳定的运行态：

- 有的设备需要 **自动启动网络守护进程**；
- 有的设备需要 **周期性运行后台任务**；
- 有的设备需要 **日志系统** 来记录运行状态；
- 甚至需要 **systemd** 或 **sysvinit** 来进行服务管理。

没有系统服务的管理，设备无法在启动后进入正常工作状态。

------

## Buildroot 的设计定位

与通用发行版不同，Buildroot 对包和服务的处理强调：

- **静态组合**：在构建时决定好所有依赖和服务。
- **最小化原则**：只选择必要的包，避免臃肿。
- **可移植性**：所有包的配置和依赖关系，都被 Makefile 和 Kconfig 记录。

这意味着：

- 你需要在构建时明确指定应用和服务，而不是像 Debian 那样“运行时再安装”。
- 通过 Buildroot，你得到的固件就是完整且一致的运行环境。

# 6.2 包的获取与构建

------

## 引言

在 Buildroot 中，**包（package）** 是整个系统的基本构建单元。
 无论是 BusyBox、glibc，还是自定义的应用程序，它们在 Buildroot 中都是一个包。
 理解包的获取与构建流程，是掌握 Buildroot 的核心关键。

------

## 1. 包的组成

在 Buildroot 的源码树里，每个包通常包含以下两个文件：

1. **`Config.in`**

   - 定义包的配置项（Kconfig 格式）。
   - 决定该包是否出现在 `menuconfig` 中。

   示例：

   ```make
   config BR2_PACKAGE_HELLO
       bool "hello"
       help
         A simple hello world program
   ```

2. **`xxx.mk`**

   - 定义包的构建规则（GNU Make 格式）。
   - 包含源码下载地址、编译方法、安装步骤。

   示例：

   ```make
   HELLO_VERSION = 1.0
   HELLO_SITE = $(call github,example,hello,$(HELLO_VERSION))
   HELLO_LICENSE = MIT
   HELLO_LICENSE_FILES = LICENSE
   
   $(eval $(autotools-package))
   ```

这样，Buildroot 就能根据配置决定是否启用包，并且知道如何获取源码和构建。

------

## 2. 源码获取方式

Buildroot 支持多种源码获取方式，典型方式包括：

- **HTTP/FTP 下载**

  ```make
  FOO_SITE = http://example.com/downloads
  ```

- **Git 仓库**

  ```make
  FOO_SITE = git://github.com/example/foo.git
  FOO_VERSION = master
  ```

- **SVN/CVS 等旧式仓库**（不常用了）。

- **本地目录**（开发调试常用）

  ```make
  FOO_SITE = /home/user/mylib
  FOO_SITE_METHOD = local
  ```

源码会被统一下载到：

```
output/build/packagename-version/
```

------

## 3. 构建流程

每个包的构建大致分为五个阶段：

1. **下载 (download)**
   - 下载源码包，存放在 `dl/` 目录下。
2. **解压 (extract)**
   - 解压到 `output/build/` 目录。
3. **补丁 (patch)**
   - 应用 Buildroot 提供的补丁。
4. **配置 (configure)**
   - 根据工具链、依赖库进行配置。
   - 常见方式：autotools, cmake, meson, 手写 Makefile。
5. **编译 (build)**
   - 调用交叉编译器（`$(TARGET_CC)`）。
6. **安装 (install)**
   - `make install` → 安装到目标目录 `output/target/`。
   - 还会安装到 staging 目录 `output/staging/`，供其他包使用头文件和库。

------

## 4. 安装路径

Buildroot 将编译产物安装到不同区域：

- **staging/ (`output/staging/`)**
  - 包含头文件和库，供其他包在编译时使用。
  - 类似于一个 sysroot。
- **target/ (`output/target/`)**
  - 最终 rootfs 的目录。
  - 只包含运行时需要的文件（剥离符号、去掉文档）。
- **host/ (`output/host/`)**
  - 宿主机工具（如 pkg-config, genext2fs）。
  - 用于辅助交叉编译。

------

## 5. 构建方式

不同的包，可能需要不同的构建工具。Buildroot 提供了大量基础宏：

- **autotools-package**：支持 GNU autotools 的包。
- **cmake-package**：支持 CMake 的包。
- **meson-package**：支持 Meson 构建系统。
- **generic-package**：自定义 Makefile 的包。

例如：

```make
$(eval $(cmake-package))
```

表示该包使用 CMake 构建系统。

------

## 6. 包构建流程总结图

```
下载源码 → 解压 → 打补丁 → 配置 → 编译 → 安装 (staging/target)
```

------

## 小结

- Buildroot 中的 **包** 由 `Config.in` + `xxx.mk` 组成。
- 源码获取方式灵活，支持 HTTP、Git、本地目录。
- 构建流程分为 **下载 → 解压 → 补丁 → 配置 → 编译 → 安装**。
- 安装结果分为 staging（开发用）、target（rootfs）、host（宿主机工具）。
- Buildroot 通过统一的 Makefile 宏，简化了对不同构建系统的支持。

# 6.3 包的依赖与配置

------

## 引言

在实际项目中，单独的一个包往往无法独立工作。
 例如：

- **OpenSSL** 依赖于 zlib；
- **Qt** 依赖 freetype、libpng、alsa-lib 等；
- **应用程序** 依赖 C 库、内核头文件。

Buildroot 的强大之处在于，它不仅能获取和构建单个包，还能 **自动解析依赖关系**，并在最终 rootfs 中保持一致性。

------

## 1. 包的依赖关系

在 Buildroot 中，依赖关系主要分为三类：

1. **编译依赖 (Build dependencies)**

   - 某个包编译时需要另一个包的头文件或库。
   - 例如：OpenSSL 在编译时需要 zlib。
   - Buildroot 会确保 zlib 先被构建并安装到 staging。

2. **运行依赖 (Runtime dependencies)**

   - 某个包运行时需要另一个包存在。
   - 例如：Python 运行时需要 libffi。
   - Buildroot 会自动把 libffi 安装到 target。

3. **工具链依赖**

   - 包可能需要 C++、线程、浮点运算等工具链特性。

   - 这些依赖通过 `Config.in` 表达，例如：

     ```make
     depends on BR2_TOOLCHAIN_HAS_THREADS
     depends on BR2_INSTALL_LIBSTDCPP
     ```

------

## 2. 依赖声明

依赖关系在 **`xxx.mk`** 文件中声明：

- **编译依赖**

  ```make
  FOO_DEPENDENCIES = bar baz
  ```

  表示 foo 包在构建前必须先构建 bar 和 baz。

- **运行依赖**
   在 `Config.in` 中声明：

  ```make
  select BR2_PACKAGE_BAR
  ```

  表示只要启用 foo，就自动启用 bar。

- **条件依赖**

  ```make
  depends on BR2_PACKAGE_ZLIB
  ```

  表示 foo 只有在启用 zlib 时才可用。

------

## 3. 配置系统（Kconfig）

Buildroot 的配置系统基于 **Kconfig**（与 Linux 内核相同），通过 `make menuconfig` 呈现。

### 配置方式

- **bool**：开/关选项

  ```make
  config BR2_PACKAGE_FOO
      bool "foo"
  ```

- **choice**：多选一

  ```make
  choice
      prompt "C library"
      config BR2_TOOLCHAIN_BUILDROOT_UCLIBC
          bool "uClibc"
      config BR2_TOOLCHAIN_BUILDROOT_GLIBC
          bool "glibc"
  endchoice
  ```

- **depends on**：依赖条件

  ```make
  depends on BR2_TOOLCHAIN_HAS_THREADS
  ```

- **select**：自动启用依赖

  ```make
  select BR2_PACKAGE_ZLIB
  ```

### 用户体验

在 `menuconfig` 中：

- 不满足条件的选项会被灰掉。
- 启用一个包时，依赖会自动启用。

------

## 4. 配置传播机制

当用户启用某个包时，Buildroot 会：

1. **解析 Kconfig**，检查依赖是否满足。
2. **解析 `.mk` 文件**，确定编译依赖。
3. **生成依赖树**，确保构建顺序正确。

例如启用 `qt5` 时：

```
qt5 → freetype → libpng → zlib
```

Buildroot 会自动按顺序构建这些包，避免手动处理依赖。

------

## 5. 可选特性与裁剪

有些包体积庞大，但功能可裁剪：

- Qt 可裁剪掉 WebEngine。
- BusyBox 可裁剪掉 telnetd。
- Python 可裁剪掉 sqlite3 支持。

Buildroot 利用 `Config.in` 暴露这些裁剪选项，让开发者在 `menuconfig` 中选择所需功能，从而平衡 **功能性** 与 **体积**。

------

## 6. 调试依赖关系

- 查看依赖树：

  ```bash
  make foo-graph-depends
  ```

  会生成 `foo-depends.dot`，可用 `dot -Tpng` 转为图像。

- 检查反向依赖：

  ```bash
  make bar-revdep
  ```

  显示哪些包依赖 bar。

这些工具对复杂系统（如 Qt + Python + OpenSSL）非常有用。

------

## 小结

- Buildroot 的依赖机制分为 **编译依赖、运行依赖、工具链依赖**。
- 依赖通过 `.mk` 和 `Config.in` 共同声明。
- 配置系统基于 Kconfig，支持 `depends on` 和 `select`。
- Buildroot 自动构建依赖树，开发者无需手动处理依赖顺序。
- 可通过 `menuconfig` 进行裁剪，实现功能和体积的平衡。

# 6.4 创建自定义包

------

## 引言

Buildroot 自带了大量软件包，但在实际项目中，我们经常需要：

- 添加自研应用程序；
- 集成第三方库或驱动；
- 定制一些开源软件的特殊版本。

这时就需要自己编写一个 Buildroot 包。

------

## 1. 包的基本结构

在 Buildroot 中，每个包通常包含两个文件：

1. **`Config.in`**（包的配置选项）
   - 定义在 `menuconfig` 中如何显示。
   - 决定包是否启用，以及依赖关系。
2. **`xxx.mk`**（包的构建规则）
   - 定义源码位置、构建方式、安装方法。

通常目录结构如下：

```
package/
├── hello/
│   ├── Config.in
│   └── hello.mk
```

------

## 2. Config.in 示例

假设我们要创建一个简单的 Hello World 应用。

```make
config BR2_PACKAGE_HELLO
    bool "hello"
    help
      A simple Hello World program.
```

- `BR2_PACKAGE_HELLO` 是 Kconfig 选项名。
- `bool "hello"` 表示它是一个开关选项。
- `help` 提供说明。

这样，在 `make menuconfig` 时，就能看到 **hello** 选项。

------

## 3. hello.mk 示例

```make
HELLO_VERSION = 1.0
HELLO_SITE = $(TOPDIR)/package/hello/src
HELLO_SITE_METHOD = local
HELLO_LICENSE = GPL-2.0+
HELLO_LICENSE_FILES = COPYING

define HELLO_BUILD_CMDS
    $(TARGET_CC) $(TARGET_CFLAGS) -o $(@D)/hello $(@D)/hello.c
endef

define HELLO_INSTALL_TARGET_CMDS
    $(INSTALL) -D -m 0755 $(@D)/hello $(TARGET_DIR)/usr/bin/hello
endef

$(eval $(generic-package))
```

- **HELLO_VERSION**：版本号。
- **HELLO_SITE**：源码来源（这里是本地目录 `src/`）。
- **HELLO_BUILD_CMDS**：编译规则，调用交叉编译器。
- **HELLO_INSTALL_TARGET_CMDS**：安装到目标 rootfs。
- `$(eval $(generic-package))`：调用 Buildroot 的通用包框架。

------

## 4. 源码文件

在 `package/hello/src/` 下放置一个简单程序：

```c
#include <stdio.h>

int main(void) {
    printf("Hello, Buildroot!\n");
    return 0;
}
```

------

## 5. 集成到 Buildroot

最后，需要把包加到 Buildroot 的全局菜单中：

1. 编辑 `package/Config.in`，添加：

   ```make
   source "package/hello/Config.in"
   ```

2. 在 `make menuconfig` 中启用：

   ```
   Target packages  --->
       hello
   ```

3. 执行编译：

   ```bash
   make hello
   make
   ```

完成后，`hello` 程序会出现在 rootfs：

```
/usr/bin/hello
```

------

## 6. 常见扩展

- **支持 autotools/cmake**

  - 如果源码用 autotools，直接写：

    ```make
    $(eval $(autotools-package))
    ```

  - 如果是 CMake：

    ```make
    $(eval $(cmake-package))
    ```

- **外部包机制**

  - 可以把自定义包放在 Buildroot 树外面，通过 `BR2_EXTERNAL` 引入。
  - 更适合团队协作，避免污染 Buildroot 主树。

------

## 7. 创建自定义包流程总结

```
编写 Config.in → 编写 xxx.mk → 添加源码 → 集成到全局 Config.in → 编译测试
```

------

## 小结

- 每个 Buildroot 包由 **Config.in**（配置）和 **xxx.mk**（构建规则）组成。
- 通过 `generic-package`、`autotools-package`、`cmake-package` 等宏，可以快速适配不同构建系统。
- 最推荐的方式是 **外部包 (BR2_EXTERNAL)**，这样可以保持 Buildroot 树干净，方便维护。

# 6.5 系统服务管理

------

## 引言

文件系统准备好之后，设备启动时还需要 **初始化进程（init system）** 来负责：

- 挂载文件系统；
- 启动网络、日志、守护进程；
- 管理后台任务；
- 提供登录终端。

在 Buildroot 中，系统服务的管理方式是 **可选的**，开发者可以根据设备复杂度选择合适的 init 系统。

------

## 1. Init 系统的类型

### (1) BusyBox init（默认）

- 最轻量级的 init 系统，直接使用 BusyBox 提供的 `init`。
- 配置文件：`/etc/inittab`。
- 优点：体积小，配置简单。
- 缺点：功能有限，不支持依赖管理、服务监控。
- 适合：小型设备、路由器、开发板。

### (2) SysV init

- 传统 Linux 发行版的 init 系统。
- 启动脚本放在 `/etc/init.d/`，通过符号链接决定启动顺序。
- 优点：成熟稳定，支持服务依赖。
- 缺点：脚本复杂，启动慢。
- 适合：中型嵌入式系统。

### (3) systemd

- 现代化的 init 系统，很多桌面/服务器发行版默认使用。
- 特点：服务并行启动、依赖管理、日志集成（journald）。
- 配置文件：`.service` 单元文件。
- 优点：功能强大，支持热插拔和资源控制。
- 缺点：体积大，复杂度高。
- 适合：大型嵌入式系统（车机、智能网关）。

------

## 2. BusyBox init 配置示例

`/etc/inittab` 示例：

```bash
# 系统启动时执行
::sysinit:/etc/init.d/rcS

# 启动 getty 进程，提供串口登录
ttyS0::respawn:/sbin/getty -L ttyS0 115200 vt100

# Ctrl-Alt-Del 时执行重启
::ctrlaltdel:/sbin/reboot
```

- `::sysinit`：指定启动脚本。
- `ttyS0::respawn`：保持串口终端。
- `::ctrlaltdel`：响应键盘事件。

------

## 3. SysV init 示例

- 启动脚本目录：`/etc/init.d/`
- 运行级别目录：`/etc/rc.d/rcN.d/`

一个典型服务脚本：

```bash
#!/bin/sh
case "$1" in
  start)
    echo "Starting myservice..."
    /usr/bin/myservice &
    ;;
  stop)
    echo "Stopping myservice..."
    killall myservice
    ;;
  restart)
    $0 stop
    $0 start
    ;;
esac
```

在 rcS 中调用：

```bash
/etc/init.d/myservice start
```

------

## 4. systemd 示例

在 `/etc/systemd/system/myservice.service` 中定义：

```ini
[Unit]
Description=My Custom Service
After=network.target

[Service]
ExecStart=/usr/bin/myservice
Restart=always

[Install]
WantedBy=multi-user.target
```

启用服务：

```bash
systemctl enable myservice
```

启动服务：

```bash
systemctl start myservice
```

------

## 5. Buildroot 中的配置方法

- 选择 init 系统：
   在 `make menuconfig` → **System configuration → Init system**：
  - BusyBox
  - SysV init
  - systemd
  - None（自定义 init 脚本）
- 添加服务：
  - 对于 BusyBox：在 overlay 中添加 `/etc/inittab`。
  - 对于 SysV：在 overlay 中放置 `/etc/init.d/myscript`。
  - 对于 systemd：在 overlay 中放置 `*.service` 文件。

------

## 6. 调试与验证

- 查看系统日志：
  - BusyBox：`dmesg` / `/var/log/messages`
  - systemd：`journalctl -xe`
- 确认服务是否运行：
  - `ps | grep myservice`
  - `systemctl status myservice`（systemd）
- 启动失败排查：
  - 检查依赖是否缺失（如缺少网络、库文件）。
  - 检查 inittab 或 service 文件是否写错路径。

------

## 7. Init 系统对比表

| Init 系统    | 配置方式        | 体积 | 功能 | 启动速度 | 适用场景       |
| ------------ | --------------- | ---- | ---- | -------- | -------------- |
| BusyBox init | `/etc/inittab`  | 极小 | 基础 | 快       | 小型设备       |
| SysV init    | `/etc/init.d/`  | 中等 | 稳定 | 较慢     | 工控、传统系统 |
| systemd      | `.service` 单元 | 较大 | 完整 | 并行     | 车机、大型系统 |

------

## 小结

- Buildroot 提供三种常见 init 系统：BusyBox、SysV init、systemd。
- 对于小型嵌入式设备，推荐 **BusyBox init**；
- 对于需要多服务依赖的系统，可以选 **SysV init**；
- 对于大型复杂设备，建议使用 **systemd**。
- 服务管理通过 overlay 或脚本集成，可以灵活适配项目需求。

# 6.6 调试包与服务

------

## 引言

在 Buildroot 编译完成后，rootfs 中应该包含我们选择的所有包和服务。
 然而，实际情况可能出现：

- 包未被正确安装；
- 库依赖缺失，程序无法运行；
- 服务未随系统启动；
- systemd 或 inittab 配置错误。

因此，调试与验证过程非常关键。

------

## 1. 验证包是否安装

### (1) 检查 rootfs 内容

在宿主机上挂载 rootfs：

```bash
sudo mount -o loop output/images/rootfs.ext4 /mnt
ls /mnt/usr/bin | grep myapp
```

- 如果缺少目标文件，说明包没有被正确安装。

### (2) 使用 Buildroot 工具

- `make myapp-rebuild`
  - 重新编译并安装包。
- `make myapp-dirclean`
  - 清除旧构建结果，确保重新构建。
- `make myapp-show-depends`
  - 显示包的依赖关系，确认是否缺少依赖。

------

## 2. 动态库依赖检查

在目标系统上，运行：

```bash
ldd /usr/bin/myapp
```

输出应指向 `/lib` 或 `/usr/lib` 下的库。

- 如果提示：

  ```
  not found
  ```

  说明库未安装，需要检查依赖声明。

在宿主机交叉工具链环境中也可验证：

```bash
arm-linux-gnueabihf-readelf -d output/target/usr/bin/myapp | grep NEEDED
```

------

## 3. 服务调试

### (1) BusyBox init

- 检查 `/etc/inittab` 是否包含服务启动项：

  ```bash
  ::respawn:/usr/bin/myservice
  ```

- 如果未启动，可以手动运行：

  ```bash
  /usr/bin/myservice
  ```

### (2) SysV init

- 检查 `/etc/init.d/myscript` 是否存在且可执行。

- 确认脚本有执行权限：

  ```bash
  chmod +x /etc/init.d/myscript
  ```

- 手动测试：

  ```bash
  /etc/init.d/myscript start
  ```

### (3) systemd

- 检查服务状态：

  ```bash
  systemctl status myservice
  ```

- 查看日志：

  ```bash
  journalctl -u myservice
  ```

- 如果服务未启用：

  ```bash
  systemctl enable myservice
  systemctl start myservice
  ```

------

## 4. 常见调试方法

- **strace 跟踪**

  ```bash
  strace /usr/bin/myapp
  ```

  可以定位缺失文件、权限问题。

- **gdb 调试**
   在目标上交叉编译 gdbserver：

  ```bash
  gdbserver :1234 /usr/bin/myapp
  ```

  在宿主机用 gdb 远程连接。

- **日志分析**

  - BusyBox：检查 `/var/log/messages` 或 `dmesg`。
  - systemd：使用 `journalctl` 查看系统日志。

------

## 5. 常见问题与排查

| 问题                  | 可能原因                    | 解决方法                                |
| --------------------- | --------------------------- | --------------------------------------- |
| 程序缺失              | `xxx.mk` 未正确定义安装步骤 | 检查 `XXX_INSTALL_TARGET_CMDS`          |
| 库缺失                | 依赖未声明                  | 在 `.mk` 添加 `XXX_DEPENDENCIES`        |
| 服务未启动            | init 配置错误               | 检查 `inittab`/`init.d`/`.service` 文件 |
| `Illegal instruction` | 工具链架构不符              | 检查 `BR2_ARM_CPU` / `BR2_x86_CPU`      |
| 启动卡死              | 循环 respawn                | 修正 inittab 配置，避免脚本异常退出     |

------

## 6. 调试流程总结图

```
包缺失 → 检查 rootfs 与安装命令
库缺失 → ldd / readelf 分析依赖
服务异常 → 检查 inittab/init.d/systemd 配置
运行异常 → strace/gdb 日志分析
```

------

## 小结

- 验证包安装可通过 `ls`、`make xxx-show-depends` 等方式。
- 库依赖问题可通过 `ldd` 和 `readelf` 检查。
- 服务调试依赖 init 系统：BusyBox → inittab，SysV → init.d 脚本，systemd → `.service`。
- `strace`、`gdb`、日志分析是定位问题的常见手段。

# 6.7 小结

------

## 包与服务的地位

如果说 **工具链和内核** 决定了嵌入式系统的“骨架”，**文件系统** 提供了“基础框架”，那么 **包与服务** 就是系统真正的“血肉与灵魂”。

- **包** 让系统获得特定功能（库、工具、应用）。
- **服务** 让这些功能在系统运行时自动化、可控化。

没有包，系统只有一个空壳；没有服务，系统无法进入正常的工作状态。

------

## 本章内容回顾

1. **引言 (6.1)**
   - 包与服务在嵌入式系统中的意义。
   - Buildroot 以“静态组合 + 全量构建”为设计哲学。
2. **包的获取与构建 (6.2)**
   - 包由 `Config.in`（配置）和 `.mk`（构建规则）组成。
   - 获取方式支持 HTTP、Git、本地目录。
   - 构建流程：下载 → 解压 → 补丁 → 配置 → 编译 → 安装。
   - 安装路径分为 `staging/`（开发）、`target/`（rootfs）、`host/`（宿主机工具）。
3. **包的依赖与配置 (6.3)**
   - 依赖分为 **编译依赖**、**运行依赖**、**工具链依赖**。
   - `depends on` 与 `select` 控制依赖关系。
   - `make xxx-graph-depends` 可生成依赖图。
4. **创建自定义包 (6.4)**
   - 最小包由 `Config.in` + `xxx.mk` 组成。
   - 通过 `generic-package`、`autotools-package`、`cmake-package` 宏快速构建。
   - 推荐使用 **外部包 (BR2_EXTERNAL)**，保持 Buildroot 树干净。
5. **系统服务管理 (6.5)**
   - 支持三种 init 系统：BusyBox init、SysV init、systemd。
   - BusyBox init 轻量、systemd 功能强大，选择取决于系统规模。
   - 服务定义方式：`inittab`、`/etc/init.d/`、`.service`。
6. **调试包与服务 (6.6)**
   - 验证包安装 → 检查 rootfs 与安装步骤。
   - 库依赖问题 → 使用 `ldd`/`readelf`。
   - 服务调试 → 根据 init 系统检查配置。
   - 常用调试工具：`strace`、`gdb`、`journalctl`。

## 对比总结

| 方面     | Buildroot               | Yocto                    |
| -------- | ----------------------- | ------------------------ |
| 包获取   | 简单（HTTP/Git）        | 复杂（多层 recipe）      |
| 包管理   | 无运行时包管理          | 支持 dpkg/rpm/opkg       |
| 依赖解析 | 基于 Kconfig + Makefile | 基于 BitBake，功能更强大 |
| 自定义包 | 简单，适合快速开发      | 复杂，但可扩展性更强     |
| 服务管理 | BusyBox/SysV/systemd    | 同样支持，但更依赖元数据 |

------

## 设计哲学总结

- **Buildroot 的定位**：轻量、快速、确定性强。
- **包管理方式**：在构建时决定功能组合，避免运行时动态安装。
- **服务管理方式**：选择合适的 init 系统，保持功能与体积的平衡。

这意味着：

- 小型系统 → 用 BusyBox init，保持 rootfs 极简。
- 中型系统 → 用 SysV init，获得传统脚本式管理。
- 大型复杂系统 → 用 systemd，支持并行启动与依赖控制。
