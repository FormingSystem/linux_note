# 10.2 文件系统镜像类型

------

## 1. 背景

在第 7 部分我们已经看到，Buildroot 可以生成不同格式的 rootfs 镜像。
 但这些格式并不是等价的，它们适合不同的 **存储介质** 和 **使用场景**。

选择正确的文件系统类型，能显著影响：

- 存储空间利用率
- 系统启动速度
- 数据可靠性
- 写入寿命

因此，本节重点对比 **常见 rootfs 镜像类型**。

------

## 2. 常见文件系统类型

### (1) **ext2 / ext3 / ext4**

- **特点**：
  - 传统 Linux 文件系统，功能完整。
  - ext3/4 支持 journaling（日志），提高可靠性。
  - ext4 支持更大容量和更多特性。
- **适用场景**：
  - eMMC、SD 卡、SATA、USB 存储。
  - 需要可写 rootfs 的系统。
- **限制**：
  - 占用空间相对较大；
  - 日志写入会增加闪存磨损。

------

### (2) **SquashFS**

- **特点**：
  - 只读、压缩文件系统。
  - 压缩率高，适合存储大规模静态内容。
- **适用场景**：
  - 固件分发（只读 rootfs）；
  - 节省空间的设备，例如路由器、IoT。
- **限制**：
  - 不支持写入，必须结合 overlayfs/jffs2 提供可写层。

------

### (3) **JFFS2 (Journaling Flash File System v2)**

- **特点**：
  - 针对原始 NAND/NOR 闪存设计。
  - 支持日志与压缩。
  - 动态调整坏块。
- **适用场景**：
  - 小容量 NAND/NOR Flash。
- **限制**：
  - 挂载速度随存储容量线性变慢（大容量 flash 启动慢）。
  - 已逐渐被 UBIFS 取代。

------

### (4) **UBIFS (Unsorted Block Image File System)**

- **特点**：
  - 针对大容量 NAND Flash 优化。
  - 基于 UBI 层，支持磨损均衡和坏块管理。
  - 性能和可靠性优于 JFFS2。
- **适用场景**：
  - 大于 256MB 的 NAND Flash 存储。
- **限制**：
  - 需要底层 UBI 支持，配置更复杂。

------

### (5) **cpio (Initramfs)**

- **特点**：
  - 简单的归档格式，不是真正的文件系统。
  - 可直接作为内核的 initramfs。
- **适用场景**：
  - 内核内置 rootfs；
  - 早期引导阶段使用（SPL/TFTP 启动）。
- **限制**：
  - 不支持写入；
  - 只适合小型 rootfs。

------

### (6) **tar**

- **特点**：
  - 归档格式，通常作为分发用。
  - 可以在宿主机解压到目标设备。
- **适用场景**：
  - 调试、NFS 挂载开发。
- **限制**：
  - 不能直接作为 rootfs，需要展开到存储介质。

------

### (7) **其他格式**

- **ISO9660**：用于光盘镜像（很少用于嵌入式）。
- **cramfs**：旧的只读压缩文件系统，已被 SquashFS 取代。

------

## 3. 对比表

| 文件系统     | 读写性 | 压缩 | 典型存储介质  | 优点                 | 缺点                |
| ------------ | ------ | ---- | ------------- | -------------------- | ------------------- |
| **ext2**     | 可写   | 否   | SD/eMMC/USB   | 简单稳定             | 无日志，崩溃恢复差  |
| **ext3**     | 可写   | 否   | SD/eMMC/USB   | 日志增强可靠性       | 占用空间大          |
| **ext4**     | 可写   | 否   | SD/eMMC/USB   | 支持大容量，特性多   | 日志影响闪存寿命    |
| **SquashFS** | 只读   | 是   | NAND/SD/eMMC  | 压缩率高，节省空间   | 只读，需要 overlay  |
| **JFFS2**    | 可写   | 是   | 原始 NAND/NOR | 适合小容量闪存       | 大容量挂载慢        |
| **UBIFS**    | 可写   | 是   | 原始 NAND     | 适合大 NAND，可靠    | 配置复杂，需 UBI    |
| **cpio**     | 只读   | 可选 | 内存          | 简单，适合 initramfs | 功能有限            |
| **tar**      | 可写   | 可选 | 任意          | 易于分发、解压       | 不能直接用作 rootfs |

------

## 4. 选择建议

- **开发调试阶段**：
  - 推荐使用 `tar` 或 `ext2`，方便修改和快速测试。
- **量产设备**：
  - 如果 rootfs 需要可写 → `ext4`（SD/eMMC）或 `UBIFS`（NAND）。
  - 如果 rootfs 主要只读 → `SquashFS` + overlayfs。
- **小型系统**：
  - 只需要最小 rootfs → `cpio` 内置进内核，适合 initramfs 启动。

------

## 小结

- Buildroot 支持多种 rootfs 镜像格式，每种都有明确的适用场景。
- 选择文件系统时要考虑 **存储介质类型、容量、是否需要可写、是否注重空间压缩**。
- 常见组合：
  - **SD/eMMC → ext4**
  - **NAND → UBIFS**
  - **只读固件 → SquashFS + overlayfs**
  - **早期启动 → cpio (initramfs)**

# 10.3 genimage 工具

------

## 1. 背景

在前一节我们看到，Buildroot 能生成各种格式的 **rootfs 镜像**（ext4、ubifs、squashfs 等）。
 但在实际设备中，仅有 rootfs 还不够。一个可启动的镜像往往需要：

- **Bootloader**（U-Boot/SPL）放在特定偏移；
- **内核镜像和设备树**放在 boot 分区；
- **Rootfs 文件系统**放在独立分区；
- **分区表**（MBR/GPT）确保系统能识别。

手动拼接这些内容既繁琐又容易出错。
 为此，Buildroot 集成了 **genimage** 工具，它能根据配置文件自动生成完整的系统镜像。

------

## 2. genimage 的工作方式

genimage 的核心思想是：

- **用户提供配置文件 (`genimage.cfg`)**，描述分区布局、文件系统类型和镜像来源；
- genimage 根据配置：
  1. 创建空的磁盘镜像文件；
  2. 在其中写入分区表；
  3. 将 Bootloader、内核、rootfs 镜像放入相应分区；
  4. 最终输出一个可烧录到 SD/eMMC/NAND 的镜像。

------

## 3. 配置文件示例

以 **i.MX6ULL 开发板 SD 卡镜像**为例：

```ini
image sdcard.img {
  hdimage {
    partition-table = "msdos"
  }

  # U-Boot 分区
  partition u-boot {
    in-partition-table = "no"
    image = "u-boot-dtb.imx"
    offset = 1024      # 偏移 1KB
  }

  # Boot 分区 (FAT)
  partition boot {
    partition-type = 0xC
    bootable = "yes"
    image = "boot.vfat"
    size = 16M
  }

  # Rootfs 分区 (ext4)
  partition rootfs {
    partition-type = 0x83
    image = "rootfs.ext4"
    size = 128M
  }
}
```

说明：

- `hdimage` 定义整张镜像的分区表类型（msdos/gpt）。
- `partition u-boot`：直接写入二进制文件，不进入分区表。
- `partition boot`：创建 FAT 分区，放置内核和设备树。
- `partition rootfs`：创建 ext4 分区，存放 rootfs。

最终生成的 `sdcard.img` 可以直接用 `dd` 或量产工具写入 SD 卡。

------

## 4. genimage 与 Buildroot 的集成

在 Buildroot 中，genimage 通常通过 **post-image 脚本**调用：

路径：

```
board/<vendor>/<board>/genimage.cfg
board/<vendor>/<board>/post-image.sh
```

`post-image.sh` 示例：

```bash
#!/bin/sh
GENIMAGE_CFG="board/freescale/common/imx/genimage.cfg"
GENIMAGE_TMP="output/build/genimage.tmp"

rm -rf "${GENIMAGE_TMP}"
mkdir -p "${GENIMAGE_TMP}"

genimage \
    --rootpath "output/target" \
    --tmppath "${GENIMAGE_TMP}" \
    --inputpath "output/images" \
    --outputpath "output/images" \
    --config "${GENIMAGE_CFG}"
```

这样，在 `make` 完成后，会自动执行 genimage，生成 `sdcard.img`。

------

## 5. genimage 的关键参数

- `--rootpath`
   指向目标根文件系统目录（通常是 `output/target`）。
- `--inputpath`
   输入目录，存放已有的 bootloader/kernel/rootfs 镜像（`output/images`）。
- `--outputpath`
   最终镜像的输出路径。
- `--config`
   指定配置文件路径。
- `--tmppath`
   临时目录，用于构建过程中的中间文件。

------

## 6. 常见应用场景

1. **SD 卡镜像**
   - Bootloader + FAT 分区 (kernel/dtb) + ext4 rootfs。
2. **eMMC 镜像**
   - 类似 SD 卡，但可能带有 boot0/boot1 分区。
3. **NAND 镜像**
   - 使用 UBI/UBIFS，genimage 可定义 `ubi` 类型分区。
4. **多分区系统**
   - 例如 Android/Linux 双系统，可在配置中定义多个 rootfs 分区。

------

## 7. 示例对比

| 场景                      | 分区配置                              | 产物         |
| ------------------------- | ------------------------------------- | ------------ |
| IoT 设备（只读 rootfs）   | U-Boot + boot(FAT) + rootfs(SquashFS) | `sdcard.img` |
| 工控板（可写 rootfs）     | U-Boot + boot(FAT) + rootfs(ext4)     | `emmc.img`   |
| 消费级产品（大容量 NAND） | U-Boot + ubi(rootfs UBIFS)            | `nand.img`   |

------

## 8. 小结

- **genimage** 是 Buildroot 生成完整系统镜像的核心工具。
- 它通过 `genimage.cfg` 描述分区和文件系统布局。
- 与 `post-image.sh` 脚本结合，可实现自动化镜像生成。
- 不同的存储介质（SD/eMMC/NAND）都能通过配置灵活适配。

# 10.4 post-image 脚本

------

## 1. 背景

在 Buildroot 的编译流程中，最后一步是 **生成目标镜像**（rootfs、boot 分区、完整 sdcard.img 等）。
 但是在实际项目中，我们常常需要额外操作，例如：

- 调用 **genimage** 生成完整分区镜像；
- 拷贝额外的固件、配置文件、校验和文件；
- 执行签名（signing），保证镜像的安全性；
- 打包压缩，准备发布。

这些动作无法由 Buildroot 内置规则完全覆盖，因此 Buildroot 提供了 **post-image 脚本机制**，允许开发者在构建完成后插入自定义逻辑。

------

## 2. 配置 post-image 脚本

在 **board 支持包**（BSP）目录下，通常会有一个 `post-image.sh` 文件，例如：

```
board/freescale/common/imx/post-image.sh
```

在 Buildroot 配置文件（defconfig）中指定：

```make
BR2_ROOTFS_POST_IMAGE_SCRIPT="board/freescale/common/imx/post-image.sh"
```

这样，在构建完成并生成 rootfs 之后，Buildroot 会自动调用该脚本。

------

## 3. post-image 脚本示例

以下示例展示了一个典型的 `post-image.sh`：

```bash
#!/bin/sh

BOARD_DIR="$(dirname $0)"
GENIMAGE_CFG="${BOARD_DIR}/genimage.cfg"
GENIMAGE_TMP="${BUILD_DIR}/genimage.tmp"

echo ">>> Running post-image script"

# 清理临时目录
rm -rf "${GENIMAGE_TMP}"
mkdir -p "${GENIMAGE_TMP}"

# 调用 genimage 生成 sdcard.img
genimage \
    --rootpath "${TARGET_DIR}" \
    --tmppath "${GENIMAGE_TMP}" \
    --inputpath "${BINARIES_DIR}" \
    --outputpath "${BINARIES_DIR}" \
    --config "${GENIMAGE_CFG}"
```

### 脚本解读

- `BOARD_DIR`：指向当前板级目录。
- `GENIMAGE_CFG`：配置文件路径。
- `TARGET_DIR`：指向 rootfs 内容（`output/target`）。
- `BINARIES_DIR`：指向镜像目录（`output/images`）。
- `BUILD_DIR`：指向构建目录（`output/build`）。
- `genimage`：根据配置文件生成最终的 `sdcard.img`。

------

## 4. 多个脚本的情况

Buildroot 允许指定多个 post-image 脚本：

```make
BR2_ROOTFS_POST_IMAGE_SCRIPT="board/foo/post-image.sh board/bar/post-process.sh"
```

它们会按顺序依次执行。
 这种方式适合 **分层处理**：

- 第一个脚本生成基础镜像；
- 第二个脚本进行签名、压缩；
- 第三个脚本生成 OTA 包。

------

## 5. 常见应用场景

| 场景         | 实现方式                                 |
| ------------ | ---------------------------------------- |
| 生成完整镜像 | 调用 genimage，输出 sdcard.img/nand.img  |
| 增加额外文件 | `cp extra.bin ${BINARIES_DIR}/`          |
| 添加校验和   | `sha256sum *.img > sha256sums.txt`       |
| 签名         | 调用 `openssl dgst -sign` 或厂商签名工具 |
| 压缩镜像     | `gzip -9 sdcard.img`                     |
| OTA 更新包   | 打包成 zip/tar，附带 metadata            |

------

## 6. 注意事项

1. **执行时机**
   - post-image 脚本在 `make` 的最后阶段执行，确保 rootfs 和内核镜像已经准备好。
2. **错误处理**
   - 如果脚本返回非零值，Buildroot 会停止并报错。
   - 建议在脚本开头加 `set -e`，保证一旦失败立即退出。
3. **跨平台兼容**
   - 脚本通常在 Linux 环境下执行，避免使用宿主机依赖的工具（如 GNU-specific 的 sed）。
   - 尽量依赖 Buildroot 内部提供的 host 工具。

------

## 7. 小结

- post-image 脚本是 Buildroot 的 **扩展点**，用于在构建结束后执行自定义操作。
- 最常见的用途是 **调用 genimage** 生成完整系统镜像。
- 它也可以完成签名、压缩、拷贝文件等任务。
- 通过多个脚本的组合，可以实现复杂的量产流程。

# 10.5 存储介质适配

------

## 1. 背景

不同的嵌入式设备使用不同类型的存储介质：

- 开发板 → 通常用 **SD 卡** 启动；
- 工控/消费设备 → 常用 **eMMC**；
- 路由器、IoT → 多数使用 **NAND Flash**；
- 一些小型 MCU SoC → 内置 **NOR Flash**。

存储介质的物理特性决定了镜像格式和部署方式必须有所区别。
 因此，Buildroot 提供了灵活机制（尤其是 **genimage** + **post-image 脚本**）来适配不同存储方案。

------

## 2. SD 卡

### 特点

- 常见于开发板，容量较大，接口标准化。
- 使用分区表（MBR/GPT），支持 ext4/FAT/SquashFS 等文件系统。

### 镜像生成

典型的 SD 卡镜像由 **三部分**组成：

1. **Bootloader (u-boot.imx / spl.bin)** → 写在固定偏移；
2. **Boot 分区 (FAT)** → 存放内核和设备树；
3. **Rootfs 分区 (ext4/squashfs)** → 存放用户空间。

### genimage 示例

```ini
image sdcard.img {
  hdimage {
    partition-table = "msdos"
  }

  partition u-boot {
    in-partition-table = "no"
    image = "u-boot-dtb.imx"
    offset = 1024
  }

  partition boot {
    partition-type = 0xC
    bootable = "yes"
    image = "boot.vfat"
    size = 16M
  }

  partition rootfs {
    partition-type = 0x83
    image = "rootfs.ext4"
    size = 128M
  }
}
```

------

## 3. eMMC

### 特点

- 和 SD 类似，但通常 **焊接在板子上**。
- 带有 **boot0/boot1 分区**，部分 SoC 用来存放 SPL/Bootloader。

### 镜像生成

- 通常生成一个 **完整磁盘镜像 (emmc.img)**；
- 使用 GPT 分区表，划分 boot 分区和 rootfs 分区；
- Bootloader 可能写入 eMMC 的特殊分区（与 genimage 配置有关）。

### 部署方式

- `dd if=emmc.img of=/dev/mmcblk0`
- 或者使用厂商烧录工具（如 NXP 的 mfgtool、Rockchip 的 upgrade_tool）。

------

## 4. NAND Flash

### 特点

- 原始 NAND 没有分区表，需要 **MTD 驱动管理**。
- 存在坏块，需要 **UBI 层**管理。
- 文件系统通常是 **UBIFS**。

### 镜像生成

步骤：

1. 生成 `rootfs.ubifs`（UBIFS 镜像）。
2. 使用 `ubinize` 工具生成 `nand.img`，包含 UBI 卷。

### genimage 示例

```ini
image nand.img {
  ubinize {
    min-io-size = 2048
    sub-page-size = 2048
    leb-size = 126976
    peb-size = 131072
    max-leb-count = 2048

    volume rootfs {
      image = "rootfs.ubifs"
    }
  }
}
```

------

## 5. NOR Flash

### 特点

- 容量较小，但支持 **XIP（执行在原地）**；
- 常用于存放 Bootloader 和小型 rootfs。

### 镜像生成

- 使用 **JFFS2** 或 **直接裸写**；
- 根文件系统通常很小（几 MB 以内）。

### genimage 示例

```ini
image nor.img {
  flash {
    size = 16M
    eraseblock = 64K
  }

  partition bootloader {
    image = "u-boot.bin"
    size = 512K
  }

  partition rootfs {
    image = "rootfs.jffs2"
    size = 8M
  }
}
```

------

## 6. 对比总结

| 存储类型  | 文件系统          | 特点                        | 适用场景                |
| --------- | ----------------- | --------------------------- | ----------------------- |
| **SD 卡** | ext4/FAT/SquashFS | 容量大，分区表标准          | 开发板、原型验证        |
| **eMMC**  | ext4/FAT          | 焊接式，带 boot0/boot1 分区 | 工控/消费设备           |
| **NAND**  | UBIFS             | 支持坏块管理，需 UBI        | IoT、路由器、大容量存储 |
| **NOR**   | JFFS2 / XIP       | 容量小，直接可执行          | Bootloader、小型系统    |

------

## 7. 小结

- Buildroot 通过 **genimage + post-image 脚本**，可以灵活适配不同存储介质。
- **SD/eMMC** → 传统分区镜像，常见于开发和量产。
- **NAND** → 需要 UBI/UBIFS，适合大容量闪存。
- **NOR** → 容量有限，但适合小型 rootfs 或 Bootloader。

# 10.6 部署与烧录

------

## 1. 背景

在前面几节，我们已经看到了 Buildroot 如何生成适配不同存储介质的镜像（SD、eMMC、NAND、NOR）。
 但生成镜像只是第一步，接下来还需要 **将镜像部署到实际硬件**，这才是真正让设备启动的关键。

部署方式因存储介质和硬件平台而异，大体分为：

1. **物理写入**（如 `dd` 或烧录工具）。
2. **在线下载**（通过 USB、TFTP、UART）。
3. **量产工具**（适合工厂批量烧录）。

------

## 2. SD 卡部署

最常见的方式是将 `sdcard.img` 写入 SD 卡：

```bash
sudo dd if=output/images/sdcard.img of=/dev/sdX bs=1M conv=fsync status=progress
```

说明：

- `if=` 输入文件，即 Buildroot 生成的镜像。
- `of=` 输出设备，通常是 `/dev/sdX`（一定要确认设备号，避免覆盖 PC 硬盘）。
- `bs=1M` 提高写入速度。
- `conv=fsync` 确保数据完全写入。

验证：

```bash
sudo fdisk -l /dev/sdX
```

可以看到分区表是否符合 `genimage.cfg` 定义。

------

## 3. eMMC 部署

eMMC 通常是板载存储，烧录方式多样：

1. **SD 卡引导 + 脚本安装**
   - 先用 SD 卡启动一个 Linux 系统；
   - 在系统中运行 `dd` 或 `parted` 将镜像写入 `/dev/mmcblkX`。
2. **厂商工具**
   - **NXP**: 使用 **mfgtool** 或 **uuu (Universal Update Utility)**。
   - **Rockchip**: 使用 **upgrade_tool** 或 **rkdeveloptool**。
   - **Allwinner**: 使用 **PhoenixSuit**。

这些工具通常支持 **USB OTG 下载模式**，直接把 PC 上的镜像刷到 eMMC。

------

## 4. NAND Flash 部署

NAND 由于存在坏块，不能直接用 `dd` 写入，需要通过 MTD/UBI 工具。

典型流程：

1. 启动到 U-Boot。
2. 使用命令写入：

```bash
nand erase.chip
nand write 0x42000000 0x0 ${filesize}
```

其中：

- `0x42000000` 是内存加载地址；
- `0x0` 是 NAND 起始地址。

1. 在 Linux 系统中，也可以用：

```bash
flash_erase /dev/mtd0 0 0
nandwrite -p /dev/mtd0 rootfs.ubi
```

------

## 5. NOR Flash 部署

NOR 容量小，一般用来存放 Bootloader 或小型 rootfs。

- 在 U-Boot 下：

  ```bash
  sf probe 0
  sf erase 0 0x100000
  sf write 0x42000000 0x0 ${filesize}
  ```

- 在 Linux 下：

  ```bash
  flashcp u-boot.bin /dev/mtd0
  ```

------

## 6. 网络/USB 下载

在开发调试阶段，很多板子支持通过 **TFTP/USB 串口**加载镜像：

- **TFTP**（U-Boot 命令）：

  ```bash
  tftp 0x42000000 zImage
  bootz 0x42000000 - 0x43000000
  ```

- **USB 下载**（视平台而定）：

  - NXP i.MX 系列：`uuu`
  - Rockchip：`rkdeveloptool`
  - STM32MP1：`STM32CubeProgrammer`

这些方法不需要 SD/eMMC，适合快速迭代。

------

## 7. 工厂量产烧录

在量产阶段，往往需要 **大规模自动化烧录**：

- **多路烧录器**：一次写入多块板子。
- **厂商 SDK 工具**：支持图形化批量操作。
- **脚本化流程**：结合 Buildroot 的 post-image 脚本生成校验和（`sha256sum`）、签名、压缩包，保证出厂一致性。

------

## 8. 部署方法对比

| 存储介质     | 常见部署方法                  | 优点           | 缺点         |
| ------------ | ----------------------------- | -------------- | ------------ |
| **SD 卡**    | `dd` 写入                     | 简单，开发方便 | 不适合量产   |
| **eMMC**     | SD 卡引导安装 / USB 工具      | 稳定，量产可用 | 依赖厂商工具 |
| **NAND**     | U-Boot nand 命令 / MTD 工具   | 支持坏块管理   | 操作复杂     |
| **NOR**      | U-Boot `sf write` / `flashcp` | 直接写入       | 容量小       |
| **网络/USB** | TFTP/uuu/rkdeveloptool        | 快速调试       | 需 PC 支持   |
| **量产**     | 多路烧录器 + 脚本             | 高效稳定       | 设备成本高   |

------

## 9. 小结

- **开发调试**：SD 卡写入、TFTP、USB 工具 → 灵活高效。
- **量产阶段**：eMMC/NAND + 厂商工具 + 批量烧录器 → 稳定可靠。
- Buildroot 生成的镜像需要根据 **存储介质** 和 **量产需求**选择合适的烧录方式。
- 推荐在 post-image 脚本中自动生成 **校验和、签名文件**，保证部署一致性。
